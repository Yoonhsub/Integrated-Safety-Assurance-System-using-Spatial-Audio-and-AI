# Semantic Audio Layering 기반 버스 탑승 보조 시스템 정리

## 1. 주제 정의

**Semantic Audio Layering 기반 버스 탑승 보조 시스템**은 시각장애인과 저시력 사용자가 버스를 탈 때, 단순히 “버스가 어느 방향에 있다”만 알려주는 것이 아니라 **버스 문, 장애물, 위험 구역처럼 서로 다른 의미를 가진 객체를 각기 다른 소리로 구분해 안내하는 시스템**이다.

핵심은 공간음향을 단순한 방향 안내음이 아니라, **목표물과 위험물을 구분하는 보조 감각**으로 사용하는 것이다.

---

## 2. 핵심 아이디어

| 구분 | 사용자에게 가지는 의미 | 소리 표현 예시 |
|---|---|---|
| 버스 문 | 따라가야 하는 목표물 | 맑고 부드러운 Ping 사운드 |
| 전동킥보드 | 피해야 하는 장애물 | 낮고 둔탁한 짧은 경고음 |
| 볼라드 | 하단 장애물 | 아래쪽 방향의 둔탁한 경고 cue |
| 잘못된 버스 | 탑승 대상이 아님 | 주의성 warning cue |
| 차도 / 위험 구역 | 즉시 멈춰야 하는 영역 | 강한 alarm cue |

이 기능의 핵심은 **소리가 단순히 위치만 알려주는 것이 아니라, “이 대상이 사용자에게 어떤 의미인지”를 알려주는 것**이다.

예를 들어 같은 오른쪽 방향에서 소리가 나더라도, 그것이 버스 문인지, 잘못된 버스인지, 장애물인지에 따라 전혀 다른 cue를 제공한다.

---

## 3. 왜 차별성이 될 수 있는가

기존 서비스들은 대부분 정보를 말로 전달하거나, 단순히 장애물을 감지하는 데 초점이 있다.

| 기존 방식 | 한계 |
|---|---|
| 버스 앱 | 도착 시간은 알 수 있지만, 실제 버스 문 위치나 탑승 순간의 위험은 알기 어려움 |
| 음성 안내 | “오른쪽으로 이동하세요”처럼 순차 안내는 가능하지만, 주변 위험물과 목표물을 동시에 표현하기 어려움 |
| 스마트 지팡이 | 전방 장애물 감지에는 강하지만, 내가 탈 버스 문이나 잘못된 버스 구분에는 한계 |
| 단순 공간음향 | 방향은 알려주지만, 그 소리가 목표물인지 위험물인지 의미 구분이 약함 |

반면 Semantic Audio Layering은 다음과 같은 차별성을 가진다.

| 차별점 | 설명 |
|---|---|
| 목표물과 위험물 구분 | 버스 문은 따라갈 소리, 장애물은 피할 소리로 다르게 표현 |
| 버스 탑승 순간에 특화 | 일반 보행이 아니라 정류장 대기 → 버스 접근 → 문 찾기 → 탑승까지의 구체적 상황에 집중 |
| 잔존 시각 보조 가능 | 흐릿하게 보이는 저시력 사용자에게 소리로 의미를 덧붙여 판단 부담을 줄임 |
| 골전도 이어폰과 궁합 | 귀를 막지 않고 현실 소리를 들으면서 가상 안내음을 추가할 수 있음 |
| 비전 AI와 공간음향의 결합 이유가 명확함 | AI가 객체 의미를 판단하고, 공간음향이 그 의미를 사용자에게 전달 |

한 문장으로 정리하면 다음과 같다.

> 이 서비스는 단순히 “버스 방향을 알려주는 앱”이 아니라, 사용자가 버스 탑승 순간에 따라가야 할 대상과 피해야 할 대상을 서로 다른 소리로 구분하게 해주는 탑승 보조 시스템이다.

---

## 4. 사용자가 제기한 의문과 답변

### 4.1 눈 감고 소리만 듣고 버스를 타는 것이 현실적인가?

이 의문은 타당하다. 따라서 이 기능은 **시각을 완전히 대체하는 기술**로 설명하면 위험하다.

더 적절한 설명은 다음과 같다.

> 공간음향은 사용자를 혼자서 완전히 버스에 태우는 기술이 아니라, 탑승 직전의 불안과 위험을 줄이는 보조 감각이다.

즉, “소리만 믿고 타라”가 아니라, **남아 있는 시각, 주변 소리, 지팡이, 안내 시스템을 함께 쓰는 구조**로 보는 것이 맞다.

---

### 4.2 시각장애인이 모두 전맹은 아닌데 도움이 될 수 있는가?

오히려 이 점 때문에 보조 감각으로서의 가치가 커질 수 있다.

많은 사용자는 완전히 안 보이는 전맹이 아니라, **흐릿하게 보이거나 일부 시야만 보이는 저시력 사용자**일 수 있다.

저시력 사용자는 버스의 큰 형체는 볼 수 있어도 다음과 같은 정보는 놓칠 수 있다.

- 버스 문이 정확히 어디 열렸는지
- 문 앞에 킥보드나 볼라드가 있는지
- 정류장 안전 구역에서 벗어났는지
- 내가 탈 버스와 다른 버스를 구분할 수 있는지

따라서 이 기능은 “시각 대체”가 아니라 **잔존 시각 보정**으로 설명하는 것이 좋다.

> 흐릿하게 보이는 장면 위에, 목표물과 위험물의 의미를 소리로 덧씌워주는 보조 시스템이다.

---

### 4.3 정류장 내 가변적인 장애물은 어떻게 판단하는가?

가변 장애물은 단순히 “있다/없다”로 판단하면 안 된다.  
현실적으로는 **위험도 기반 판단**이 필요하다.

판단 흐름은 다음과 같이 정리할 수 있다.

```text
객체 인식
→ 사용자 기준 위치 추정
→ 사용자 이동 경로와 겹치는지 판단
→ 가까운지 / 움직이는지 / 위험한 객체인지 계산
→ 위험도 높은 객체만 우선 안내
```

예시는 다음과 같다.

| 상황 | 판단 |
|---|---|
| 멀리 있는 킥보드 | 낮은 위험 |
| 버스 문 앞의 킥보드 | 높은 위험 |
| 사용자 정면 1m 앞 볼라드 | 높은 위험 |
| 옆으로 지나가는 사람 | 일시적 주의 |
| 차도 방향으로 이동 | 즉시 경고 |

중요한 점은 **모든 객체에 소리를 내지 않는 것**이다.

> 사용자의 탑승 경로와 충돌할 가능성이 높은 객체만 우선 안내한다.

---

### 4.4 소리가 너무 많으면 오히려 혼란스럽지 않은가?

맞다. 이 기능의 가장 큰 위험 요소 중 하나는 소리 과부하다.

그래서 설계 원칙은 다음처럼 잡아야 한다.

| 원칙 | 설명 |
|---|---|
| 모든 객체를 안내하지 않음 | 가까운 위험물과 탑승 목표만 안내 |
| 위험 cue 우선 | 장애물이 있으면 버스 문 안내보다 먼저 경고 |
| 짧고 구분되는 소리 | 긴 음성보다 짧은 cue 중심 |
| 의미별 음색 구분 | 목표음, 경고음, 위험음을 다르게 설계 |
| 골전도 전제 | 주변 환경음을 막지 않음 |

즉, “많은 소리를 동시에 들려준다”가 아니라 다음과 같이 설명해야 한다.

> 필요한 순간에, 가장 중요한 대상만, 의미가 다른 짧은 소리로 알려준다.

---

## 5. 서비스적으로 설득력 있는 표현

발표나 기획서에서는 다음과 같이 설명할 수 있다.

> 본 기능은 공간음향을 단순 방향 안내음으로 사용하는 것이 아니라, 객체의 의미에 따라 서로 다른 소리 질감을 부여하는 Semantic Audio Layering 방식입니다. 버스 문은 따라가야 할 맑은 목표음으로, 전동킥보드나 볼라드는 피해야 할 낮은 경고음으로 표현하여 사용자가 탑승 직전의 공간 상황을 청각적으로 이해할 수 있도록 돕습니다.

더 짧게 말하면 다음과 같다.

> 흐릿하게 보이는 장면에 소리로 의미를 덧씌워, 사용자가 버스 문과 위험물을 구분하도록 돕는 보조 감각 시스템입니다.

## 6. 월요일 발표용 핵심 메시지

발표에서는 다음 세 문장을 중심으로 잡는 것이 좋다.

1. **공간음향은 단순히 버스 방향을 알려주는 기능이 아니다.**
2. **버스 문, 장애물, 위험 구역처럼 객체의 의미를 서로 다른 소리로 구분하게 하는 기능이다.**
3. **이를 통해 전맹뿐 아니라 저시력 사용자도 남아 있는 시각 정보와 청각 cue를 함께 사용해 탑승 판단 부담을 줄일 수 있다.**

---

## 7. V3 Mock 시나리오에서 보여줄 수 있는 시연 흐름

현재 V3 Mock 시나리오 구조에서는 실제 비전 AI나 실제 3D 오디오 엔진을 바로 구현하기보다, 다음 흐름을 Mock으로 보여주는 것이 현실적이다.

```text
버스 문 감지
→ 목표음 softPing 표시

킥보드 감지
→ 위험음 lowWarning 표시

볼라드 감지
→ 하단 장애물 dullAlert 표시

목표물 + 장애물 동시 감지
→ 위험 cue 우선 표시

장애물 회피 완료
→ 다시 버스 문 목표음으로 복귀
```

이 시연을 통해 다음 메시지를 보여줄 수 있다.

> 기존 V3는 버스 접근과 지오펜스 상황을 Mock으로 시각화했다면, Semantic Audio Layering 확장은 버스 문, 킥보드, 볼라드 같은 객체의 의미에 따라 서로 다른 음향 cue를 배치하는 구조를 추가한 것이다.

## 8. 개발 방향

월요일까지의 현실적인 개발 범위는 다음과 같이 제한하는 것이 좋다.

### 구현할 것

- Mock 기반 의미 객체 모델 추가
- 버스 문 / 킥보드 / 볼라드 / 위험 구역 객체 정의
- 객체별 cue texture, priority, pan, gain 표시
- Mock 시나리오 버튼 추가
- 캔버스에 의미 객체 마커 표시
- 수치 패널에 현재 활성 cue 표시
- 발표에서 시연 가능한 흐름 구성

### 구현하지 않을 것

- 실제 비전 AI 연동
- 실제 카메라 객체 인식
- 실제 BLE/RSSI 연동
- 실제 3D 오디오 엔진 구현
- 백엔드 API 수정
- shared contracts 수정
- mobile_sensors 수정

## 10. 최종 정리

이 아이디어의 차별성은 단순히 **“3D 소리를 쓴다”**가 아니다.

진짜 차별성은 다음에 있다.

> 버스 탑승 상황에서 목표물과 위험물을 의미별로 구분하고, 그 의미를 공간음향으로 전달한다는 점.

따라서 발표와 개발 방향 모두 **“시각 대체”**가 아니라 **“잔존 시각과 청각을 결합한 보조 판단 시스템”**으로 설명하는 것이 가장 현실적이고 설득력 있다.
