# 모비 프로젝트 4월 말까지 개인별 구현 범위 수정안

## 0. 문서 작성 기준

이 문서는 `모비 (2).docx`의 월별/주차별 계획을 바탕으로, 4월 말까지 각 팀원이 **from scratch**로 구현해야 할 개발 범위를 재정리한 문서이다.

원문 계획은 유지하되, 아래 수정 사항을 반영하였다.

### 반영한 수정 사항

1. **안준환 담당 범위에서 헤드트래킹 파트 제외**
   - 헤드트래킹 센서는 추후 구매 예정이므로 4월 말까지 구현 범위에서 제외한다.
   - 따라서 `헤드트래킹 센서 통신`, `헤드트래킹 데이터를 활용한 공간음향`, `비콘 + 헤드트래킹 통합 테스트`는 4월 구현 범위에서 제외한다.
   - 단, 스마트폰 내장 나침반/방향 센서, BLE 비콘 수신, RSSI 기반 거리 계산 등 당장 구현 가능한 센서 파트는 유지한다.

2. **심현석 담당 범위에서 공공데이터 API 기반 버스 정보 연동 파트를 김도성에게 이전**
   - 실시간 버스 위치, 버스 도착 정보, 저상버스 여부 필터링 등 공공데이터 API 연동은 김도성이 담당한다.
   - 심현석은 해당 데이터를 저장하거나 앱/서버 흐름에 연결할 수 있도록 Firebase DB 구조와 백엔드 인터페이스를 준비한다.

3. **김도성 담당 범위에서 아래 테스트 계획 제거**
   - 비콘이 몇 m 거리에서 잡히는지 테스트 계획
   - 공간음향이 방향을 제대로 주는지 테스트 계획

## 1. 1~5주차에 대한 해석

문서상 4월 계획은 **5주차부터 시작**한다.  
5주차의 내용은 팀 전원 공통 계획으로, 다음 항목이다.

```txt
- 관련 정보 수집
- 오픈소스, 라이브러리 및 API 조사
- 사용 기술 학습
- 개인별 담당 개발 파트에 대한 상호 피드백
```

따라서 **1~5주차는 실질적인 개인별 개발 구현 업무가 거의 없는 준비 단계**로 보는 것이 맞다.

정리하면 다음과 같다.

| 구간 | 성격 | 개발 구현 여부 |
|---|---|---|
| 1~4주차 | 프로젝트 주제 선정, 기획, 문서화, 역할 논의 단계로 해석 | 실질 구현 없음 |
| 5주차, 4.1~4.7 | 기술 조사, 학습, 개발 준비, 역할 피드백 | 본격 구현 전 준비 단계 |
| 6주차부터 | 개인별 담당 기능 구현 시작 | 실질 구현 시작 |

따라서 이 문서에서는 **4월 말까지의 실질 개발 범위를 6~9주차 중심으로 정리**한다.

단, 9주차는 일정상 4월 29일~5월 초에 걸치므로, 4월 말 기준으로는 **착수 또는 초안 구현 범위**로 본다.

## 2. 4월 말까지 팀 전체 목표

4월의 전체 목표는 다음과 같이 정리할 수 있다.

```txt
핵심 기능 구현 및 프론트/백엔드 기초 구축
주요 UI 레이아웃 완성
센서/DB 기본 로직 구현
공공데이터 API 연동 기반 마련
```

4월 말 기준 전체 시스템은 완성형 서비스가 아니라, 아래 흐름을 부분적으로 시연할 수 있는 **기초 프로토타입**을 목표로 한다.

```txt
사용자 앱 실행
→ 음성 입력 또는 버튼 입력
→ 위치 정보 전송
→ 백엔드에서 안전/위험 판별
→ 필요 시 FCM 알림 발생
→ 공공데이터 API에서 버스 도착/저상버스 정보 조회
→ 사용자 앱에 버스 정보 표시
→ 승객 탑승 요청을 Firebase에 저장
→ 기사 앱 또는 기사 화면에서 요청 확인
```

# 4. 윤현섭 구현 범위

## 4.1 담당 역할

윤현섭은 4월 말까지 **사용자 앱의 프론트엔드 기초와 음성 인터페이스**를 구현해야 한다.

핵심 역할은 다음과 같다.

```txt
Flutter 앱 기본 구조 생성
사용자 앱 메인 화면 구현
STT 기반 음성 입력 구현
TTS 기반 음성 안내 구현
시각장애인/노약자용 접근성 UI 구현
백엔드 API 결과 화면 렌더링
기사용 앱 기본 UI 초안 작성
```

---

## 4.2 from scratch 구현 목록

### 4.2.1 Flutter 프로젝트 기본 구조 생성

최소 구조 예시는 다음과 같다.

```txt
apps/passenger_app/lib/
  main.dart
  src/
    app.dart
    pages/
      home_page.dart
    services/
      backend_api_client.dart
    widgets/

apps/driver_app/lib/
  main.dart
  src/
    app.dart
    pages/
      home_page.dart
    services/
      backend_api_client.dart
    widgets/
```

향후 추가 예정 파일 예시:

```txt
apps/passenger_app/lib/src/services/stt_service.dart
apps/passenger_app/lib/src/services/tts_service.dart
apps/passenger_app/lib/src/widgets/large_accessible_button.dart
apps/passenger_app/lib/src/widgets/bus_arrival_card.dart
apps/driver_app/lib/src/widgets/driver_request_card.dart
```

구현해야 할 것:

```txt
- 앱 실행 진입점 구성
- 사용자용 화면 라우팅 구성
- 기사용 화면 확장 가능 구조 마련
- 공통 버튼/카드 위젯 분리
- 서비스 계층 분리
```

### 4.2.2 사용자 앱 메인 화면 구현

사용자 앱은 시각장애인과 노약자를 대상으로 하므로, 복잡한 메뉴보다 단순한 구조가 중요하다.

구현해야 할 화면 요소:

```txt
- 앱 제목
- 현재 상태 안내 문구
- 초대형 음성 입력 버튼
- 버스 정보 확인 버튼
- 안전 상태 확인 버튼
- 탑승 요청 버튼
- 도움말 버튼
```

UI 조건:

```txt
- 큰 버튼
- 큰 글씨
- 고대비 색상
- 복잡한 탭 구조 최소화
- 한 화면에서 핵심 기능 접근 가능
- 아이콘만 있는 버튼 사용 금지
```

예상 화면 흐름:

```txt
앱 실행
→ "어디로 가시나요?" 화면 표시
→ "말하기 시작" 버튼
→ 음성 입력
→ 결과 텍스트 표시
→ 버스 정보 또는 탑승 요청 화면으로 이동
```

---

### 4.2.3 STT 음성 입력 기능 구현

사용자의 음성 질문을 텍스트로 변환하는 기능이다.

구현해야 할 것:

```txt
- speech_to_text 라이브러리 설치
- 마이크 권한 요청
- 음성 인식 시작
- 음성 인식 중 상태 표시
- 음성 인식 종료
- 인식 결과 텍스트 표시
- 인식 실패 처리
- 재시도 버튼 제공
```

입력 예시:

```txt
"502번 버스 언제 와?"
"충북대 정문 가고 싶어"
"다음 버스 알려줘"
"탑승 요청해줘"
```

4월 말까지 자연어 처리 전체를 완성할 필요는 없다.  
최소 목표는 **음성을 텍스트로 받아 화면에 표시하고, 이후 백엔드 요청으로 넘길 수 있는 구조를 만드는 것**이다.

---

### 4.2.4 TTS 음성 출력 기능 구현

앱이 사용자에게 안내 메시지를 음성으로 읽어주는 기능이다.

구현해야 할 것:

```txt
- flutter_tts 라이브러리 설치
- 한국어 음성 출력 확인
- 안내 문장 재생
- 재생 중 상태 표시
- 재생 완료 처리
- 재생 실패 시 텍스트 안내 유지
```

출력 예시:

```txt
"502번 버스가 약 3분 후 도착합니다."
"위험 구역에 접근 중입니다. 뒤로 물러나세요."
"탑승 요청이 기사에게 전달되었습니다."
"저상버스가 곧 도착합니다."
```

### 4.2.6 백엔드 API 결과 렌더링

백엔드에서 받은 데이터를 화면에 표시하는 기능이다.

표시해야 할 데이터:

```txt
- 현재 안전 상태
- 위험 경고 메시지
- 버스 번호
- 도착 예정 시간
- 남은 정류장 수
- 저상버스 여부
- 탑승 요청 상태
```

4월 말까지 실제 API가 완전하지 않을 수 있으므로, 다음 2가지 방식 모두 가능해야 한다.

```txt
- 실제 API 응답 렌더링
- mock JSON 응답 렌더링
```

예상 카드 예시:

```txt
502번 버스
도착 예정: 약 3분 후
저상버스: 예
혼잡도: 보통
```

# 5. 안준환 구현 범위

## 5.1 담당 역할

안준환은 4월 말까지 **스마트폰 센서와 BLE 비콘 기반의 정밀 탑승 유도 기초 기술**을 구현해야 한다.

단, 헤드트래킹 센서는 아직 구매 전이므로 4월 구현 범위에서 제외한다.

---

## 5.2 4월 구현 범위에서 제외되는 것

아래 항목은 4월 말까지 구현 대상이 아니다.

```txt
- 아두이노 등 외부 헤드트래킹 센서 통신
- 헤드트래킹 센서값 수집
- 헤드트래킹 데이터를 활용한 3D 공간음향 렌더링
- 비콘 신호와 헤드트래킹 데이터를 결합한 공간음향 통합 테스트
```

이 항목들은 센서 구매 이후 별도 단계에서 진행한다.

---

## 5.3 from scratch 구현 목록

### 5.3.1 스마트폰 내장 방향 센서 확인

헤드트래킹 센서는 제외하지만, 스마트폰 자체의 나침반 또는 방향 센서는 사용할 수 있다.

구현해야 할 것:

```txt
- 스마트폰 나침반 센서 접근
- 현재 방위각 값 읽기
- 센서 권한 또는 사용 가능 여부 확인
- 방위각 값을 패키지 내부 예제 또는 로그에 출력
```

예상 출력:

```json
{
  "headingDegrees": 132.5,
  "accuracy": "MEDIUM",
  "updatedAt": "2026-04-18T14:32:00+09:00"
}
```

이 기능은 추후 공간음향이나 방향 안내 기능에 사용할 수 있는 기초값이다.

---

### 5.3.2 BLE 비콘 수신 기능 구현

BLE 비콘을 스캔하고 주변 비콘 정보를 읽는 기능을 구현한다.

구현해야 할 것:

```txt
- flutter_blue_plus 또는 유사 BLE 라이브러리 조사 및 설치
- Bluetooth 권한 요청
- 위치 권한 요청
- 주변 BLE 기기 스캔
- 특정 비콘 ID 필터링
- 비콘 이름, UUID, RSSI 출력
- 스캔 시작/중지 처리
```

출력 예시:

```json
{
  "beaconId": "MOBI_BEACON_001",
  "rssi": -62,
  "estimatedDistanceMeters": 2.8,
  "signalLevel": "CLOSE",
  "lastDetectedAt": "2026-04-18T14:32:10+09:00"
}
```

---

### 5.3.3 RSSI 기반 거리 계산 로직 구현

BLE 비콘의 RSSI 값을 바탕으로 대략적인 거리 추정 로직을 작성한다.

구현해야 할 것:

```txt
- RSSI 값 수집
- TxPower 기준값 설정
- RSSI → 거리 추정 공식 적용
- 거리값 smoothing 처리
- 이상값 제거
- 최근 N개 값 평균 처리
```

거리 계산 예시:

```txt
RSSI -50 → 매우 가까움
RSSI -65 → 가까움
RSSI -80 → 멀어짐
RSSI -90 이하 → 신호 약함
```

정확한 미터 단위 정밀도까지 보장할 필요는 없다.  
4월 목표는 **비콘 신호 세기를 기반으로 가까움/멀어짐 정도를 판단할 수 있는 기초 로직**을 만드는 것이다.

# 7. 김도성 구현 범위

## 7.1 담당 역할

김도성은 4월 말까지 다음 두 가지 축을 담당한다.

```txt
1. 공공데이터 API 기반 버스 정보 연동
2. 2학기 AI 비전 기능을 위한 데이터 수집 계획 및 모델 리서치
```

원문 문서에서는 김도성이 AI 비전 데이터 수집 및 테스트 시나리오를 담당했지만, 수정안에서는 심현석에게 있던 **공공데이터 API 기반 실시간 버스 위치/저상버스 필터링 로직**을 김도성이 맡는다.

## 7.2 김도성 담당 범위에서 제외되는 것

아래 항목은 김도성의 4월 구현 범위에서 제외한다.

```txt
- 비콘이 몇 m 거리에서 잡히는지 테스트 계획
- 공간음향이 방향을 제대로 주는지 테스트 계획
```

즉 김도성은 비콘/공간음향의 성능 검증 계획을 작성하지 않는다.  
이 부분은 4월 말 범위에서 제외하거나 추후 별도 담당자가 정한다.

### 7.3.5 버스 혼잡도 정보 처리

혼잡도 정보가 API에서 제공되는 경우 처리한다.

구현해야 할 것:

```txt

```

표준값 예시:

```txt
LOW
NORMAL
HIGH
UNKNOWN
```
