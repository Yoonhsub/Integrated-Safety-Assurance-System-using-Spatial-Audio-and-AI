# docs/read/CONTRIBUTING.md

> MOBI 프로젝트에 기여하는 모든 팀원과 AI 에이전트는 이 문서를 반드시 따른다.  
> 이 문서는 GitHub 협업 과정에서 **브랜치 혼선, 역할 침범, AI 병렬 개발 충돌, PR 병합 문제**를 줄이기 위한 최상위 기여 규칙이다.

---

## 1. 기본 원칙

### 1.1 main 브랜치 직접 push 금지

어떤 팀원도 `main` 또는 `master` 브랜치에 직접 push하지 않는다.

모든 변경은 다음 흐름을 따른다.

```txt
개인 작업 브랜치 생성
→ 자기 담당 범위 작업
→ 문서/검증 파일 최신화
→ Pull Request 생성
→ 리뷰
→ 병합
```

예외적으로 프로젝트 초기 세팅 담당자가 최초 기준선을 올리는 경우만 직접 push가 가능하다.  
이후부터는 반드시 PR 기반으로 작업한다.

---

### 1.2 자기 담당 범위 외 수정 금지

각 팀원은 자기 담당 폴더와 문서만 수정한다.

| 팀원 | 주요 담당 범위 |
|---|---|
| 윤현섭 | `apps/passenger_app`, `apps/driver_app` |
| 심현석 | `backend/api`, `infrastructure/firebase`, 백엔드 관련 shared contract |
| 안준환 | `packages/mobile_sensors` |
| 김도성 | `services/public_data`, `ai_vision` |
| 공통 | `packages/shared_contracts`, 루트 문서, GitHub 설정, 아키텍처 문서 |

자기 담당 범위 밖의 파일을 수정해야 하는 경우, 즉시 작업을 중단하고 `docs/rw/충돌 이슈.md`에 기록한다.

---

### 1.3 문서 우선순위

이 섹션은 계약 영역별 우선순위 분리를 적용한다.

문서 간 내용이 충돌하면 `docs/read/프로젝트 4월분 개발에 관한 공통 프롬프트(AI 절대필독!).md`의 §2.1 문서 우선순위를 따른다.

```txt
[역할/범위/섹션 진행 기준]
1. docs/read/프로젝트 4월분 개발에 관한 공통 프롬프트(AI 절대필독!).md
2. 각 팀원별 에이전트 필독사항.md
3. docs/rw/선행작업의존성 정리.md
4. docs/02_4월_개인별_구현범위_수정안.md
5. docs/01_요구사항명세서.md

[API 필드명/응답 구조/enum 기준]
1. packages/shared_contracts/api/*.schema.json
2. backend/api/app/schemas/*.py
3. docs/rw/API_CONTRACTS.md
4. 각 팀원별 문서와 구현 범위 문서

[Firebase RTDB 경로/필드 기준]
1. infrastructure/firebase/realtime_database.schema.json
2. infrastructure/firebase/database.rules.json
3. docs/rw/DATA_SCHEMA.md
4. docs/01_요구사항명세서.md

[소유권/협업 규칙 기준]
1. module_ownership.json
2. .github/CODEOWNERS
3. docs/rw/MODULE_OWNERSHIP.md, docs/read/PULL_REQUEST_RULES.md, docs/read/BRANCH_STRATEGY.md, docs/read/COMMIT_CONVENTION.md
```

`docs/rw/선행작업의존성 정리.md`는 역할/범위/섹션 진행 기준의 3순위 문서이며, 작업 순서와 중단/보류 기준을 정하는 운영 규칙 문서로 본다.

위 표가 이 문서에서 사용하는 계약 영역별 우선순위의 최종 기준이며, 별도 부록 번호를 중복해서 두지 않는다.

`docs/00_원본_모비_프로젝트_계획서.docx`는 초기 기획 근거 문서이다.  
4월 실제 구현 범위는 반드시 `docs/02_4월_개인별_구현범위_수정안.md`와 팀원별 에이전트 필독사항을 우선한다.

## 2. 작업 시작 전 필수 확인

모든 팀원과 AI 에이전트는 작업 시작 전 다음 파일을 읽는다.

```txt
docs/read/AGENT_REQUIRED_READING.md
docs/read/프로젝트 4월분 개발에 관한 공통 프롬프트(AI 절대필독!).md
docs/rw/선행작업의존성 정리.md
docs/02_4월_개인별_구현범위_수정안.md
자기 팀원의 에이전트 필독사항.md
docs/rw/충돌 이슈.md
docs/rw/공통 진행사항.md
```

AI 에이전트가 작업하는 경우, 에이전트는 반드시 자신이 어느 팀원의 에이전트인지 먼저 확인해야 한다.

허용되는 에이전트 정체성:

```txt
- 심현석의 에이전트
- 윤현섭의 에이전트
- 안준환의 에이전트
- 김도성의 에이전트
```

팀원 정체성이 명시되지 않았다면 개발을 시작하지 않는다.

## 3. 브랜치 작업 원칙

상세 브랜치 전략은 `docs/read/BRANCH_STRATEGY.md`를 따른다.  
아래는 해당 문서의 핵심 요약 규칙이다.

```txt
feature/{team-member}-section-{section-number}-{short-description}
fix/{team-member}-section-{section-number}-{short-description}
docs/{team-member-or-common}-{short-description}
```

예시:

```txt
feature/hyunseok-section-02-backend-base
feature/hyunseop-section-04-passenger-ui
feature/junhwan-section-06-rssi-distance
feature/doseong-section-02-public-data-mock
docs/common-dependency-rules
```

브랜치 하나에는 가능한 한 하나의 섹션 또는 하나의 문서 변경만 담는다.

## 4. AI 에이전트 개발 절차

### 4.1 섹션 순차 수행

각 팀원의 에이전트는 자기 필독사항 문서에 명시된 12섹션을 반드시 순서대로 수행한다.

```txt
섹션 1 → 섹션 2 → 섹션 3 → 섹션 4 → 섹션 5 → 섹션 6 → 섹션 7 → 섹션 8 → 섹션 9 → 섹션 10 → 섹션 11 → 섹션 12
```

절대 이전 섹션을 건너뛰고 후속 섹션으로 점프하지 않는다.

---

### 4.2 구현 섹션과 검토 섹션 구분

```txt
섹션 2, 4, 6, 8, 10
→ 직접 구현 섹션

섹션 3, 5, 7, 9, 11
→ 직전 구현 섹션 검토/패치 섹션

섹션 12
→ 전체 심층검증 및 최종 개발 보고서 작성
```

검토/패치 섹션에서는 직전 구현 결과만 검토한다.  
문제가 있으면 해당 문제만 패치하고, 문제가 없으면 사용자에게 다음 섹션으로 넘어가도 된다고 보고한다.  
에이전트가 사용자 허락 없이 다음 구현 섹션으로 넘어가면 안 된다.

---

### 4.3 섹션 종료 후 필수 작업

각 섹션이 끝나면 반드시 다음을 수행한다.

```txt
1. 관련 문서 최신화
2. docs/rw/공통 진행사항.md의 자기 팀원 기록 공간 최신화
3. 검토/패치 섹션이면 자기 팀원 디버그 리포트 최신화
4. docs/rw/선행작업의존성 정리.md에 명시된 선행 섹션을 완료한 경우 해당 상태 최신화
5. 충돌이 있으면 docs/rw/충돌 이슈.md 기록
6. 사용자에게 GitHub 업로드 권유 문구 출력
```

섹션 종료 보고 시 반드시 아래 문구를 포함한다.

```txt
**지금 섹션 X 끝났습니다. 캡스톤 과목 점수 획득 및 충돌 병목 최소화를 위해 지금까지 진행상황 깃헙에 올리시는 것을 권유드립니다.**
```

## 5. 선행작업 의존성 처리

모든 에이전트는 자기 섹션을 시작하기 전에 `docs/rw/선행작업의존성 정리.md`를 확인한다.

### 5.1 선행 작업물이 미구현인 경우

자기 섹션이 특정 팀원의 선행 작업물에 의존하고, 해당 선행 섹션이 아직 `(미구현)`이면 다음 절차를 따른다.

```txt
1. 자기 섹션 중 선행 작업물이 필요한 하위 작업을 중단한다.
2. 선행 작업물 없이 가능한 자기 담당 내부 작업만 진행한다.
3. 선행 작업물이 반드시 필요한 시점에서는 작업을 중단한다.
4. docs/rw/충돌 이슈.md에 "선행 작업 의존성 미충족"으로 기록한다.
5. 사용자에게 선행 담당 팀원에게 먼저 작업물을 만들도록 요청하라고 보고한다.
```

사용자에게 전달할 문구:

```txt
**현재 작업은 [팀원명]의 선행 작업물인 [구체적 작업물]이 먼저 필요하므로 더 이상 진행할 수 없습니다. 반드시 [팀원명]에게 연락하여 해당 선행 작업물을 먼저 생성하도록 요청한 뒤 진행해야 합니다.**
```

---

### 5.2 선행 작업물이 구현완료로 표시된 경우

`docs/rw/선행작업의존성 정리.md`에서 선행 섹션이 `(구현완료_날짜)`로 표시되어 있더라도, 후행 에이전트는 반드시 실제 파일/코드/문서를 2차 확인한다.

확인 항목:

```txt
- 구현 파일이 실제로 존재하는가
- 문서에 명시된 필드/상태값/응답 구조가 실제 구현과 일치하는가
- 타 팀원 영역을 수정하지 않고 사용할 수 있는가
- 후행 섹션에서 필요한 최소 산출물이 준비되어 있는가
```

문제가 있으면 타 팀원 파일을 직접 수정하지 말고, `docs/rw/충돌 이슈.md`에 기록한다.

## 6. 충돌 이슈 처리

다음 상황은 반드시 `docs/rw/충돌 이슈.md`에 기록한다.

```txt
- 다른 팀원의 담당 폴더를 수정해야 하는 경우
- API 계약을 임의로 변경해야 하는 경우
- Firebase Realtime Database 스키마를 바꿔야 하는 경우
- shared contract를 임의로 변경해야 하는 경우
- 선행 작업 의존성이 미충족된 경우
- 사용자가 충돌 미해결 상태에서 강행 개발을 요청한 경우
```

충돌 이슈를 기록한 뒤에는 사용자에게 관련 팀원 이름을 명시하여 보고한다.

예시:

```txt
**현재 작업은 김도성 담당 공공데이터 표준 JSON과 충돌 가능성이 있어 더 이상 개발을 진행할 수 없습니다. 반드시 김도성에게 연락하여 이 이슈를 논의한 뒤 진행해야 합니다.**
```

## 7. 공통 진행사항 기록

모든 섹션 종료 시 `docs/rw/공통 진행사항.md`를 최신화한다.

작성 원칙:

```txt
- 자기 팀원 기록 공간만 수정한다.
- 다른 팀원 기록 공간을 수정하지 않는다.
- 완료 섹션, 수행 작업, 수정 파일, 남은 이슈, 진척도, 다음 작업을 기록한다.
- 충돌 이슈가 있으면 CONFLICT 번호를 기록한다.
- docs/rw/선행작업의존성 정리.md 관련 여부를 기록한다.
```

디버그 리포트에는 다음을 남긴다.

```txt
- 검토 대상 섹션

- 발견한 문제
- 패치 여부
- 패치한 내용
- 미해결 이슈
- 충돌 가능성

```

## 10. Pull Request 규칙

상세 PR 규칙은 `docs/read/PULL_REQUEST_RULES.md`를 따른다.  
아래는 해당 문서의 핵심 요약 규칙이다.

PR에는 반드시 다음 내용을 포함한다.

```txt
- 담당 팀원
- 사용한 AI 에이전트 여부
- 완료 섹션 번호
- 수정한 파일 목록
- 타 팀원 영역 수정 여부
- docs/rw/충돌 이슈.md 기록 여부
- docs/rw/공통 진행사항.md 최신화 여부
- 디버그 리포트 최신화 여부
- docs/rw/선행작업의존성 정리.md 확인 여부
- 테스트/검증 결과
```

PR 제목 예시:

```txt
[심현석][Section 02] FastAPI/Firebase 기본 구조 구현
[윤현섭][Section 04] 사용자 앱 메인 UI 구현
[안준환][Section 06] RSSI 거리 추정 로직 구현
[김도성][Section 02] 공공데이터 API 조사 및 mock 기준 작성
```

## 11. 커밋 규칙

상세 커밋 규칙은 `docs/read/COMMIT_CONVENTION.md`를 따른다.  
아래는 해당 문서의 핵심 요약 규칙이다.

```txt
type(scope): summary
```

허용 type:

```txt
feat      기능 추가
fix       버그 수정
docs      문서 작성/수정
refactor  구조 개선
test      테스트 추가/수정
chore     설정/기타 작업
```

예시:

```txt
feat(hyunseok): add geofence route skeleton
docs(common): add dependency management rules
fix(junhwan): adjust RSSI distance state labels
docs(doseong): add public data mock response format
```

## 12. 병합 전 체크리스트

PR 병합 전 반드시 아래를 확인한다.

```txt
[ ] main 브랜치에 직접 push하지 않았다.
[ ] 브랜치명이 작업 범위를 나타낸다.
[ ] 자기 담당 범위만 수정했다.
[ ] 다른 팀원 영역을 수정한 경우 docs/rw/충돌 이슈.md에 기록했다.
[ ] docs/rw/공통 진행사항.md를 최신화했다.
[ ] 검토/패치 섹션이면 디버그 리포트를 최신화했다.
[ ] docs/rw/선행작업의존성 정리.md를 확인했다.
```

## 13. 금지 사항

아래 행동은 금지한다.

```txt
- main 브랜치 직접 push
- 자기 담당 범위 외 파일 무단 수정
- 다른 팀원 에이전트 기록 공간 수정
- 충돌 이슈를 기록하지 않고 임의 해결
- 선행 작업물이 없는 상태에서 API/DB/mock 구조 임의 확정
- 에이전트 섹션 순서 건너뛰기
- 검토 섹션에서 다음 구현 섹션까지 임의 진행
- 해결된 충돌 이슈 삭제
- 디버그 리포트 삭제
- 최종 보고서 없이 섹션 12 완료 보고
```

## 14. 프로젝트 초기 문서 생성 단계에 대한 예외

따라서 이 단계에서는 `docs/rw/공통 진행사항.md`에 팀원별 섹션 기록을 남기지 않아도 된다.

단, 최종 패키징 시에는 `docs/read/PACKAGE_MANIFEST.txt`에 어떤 문서가 추가되었는지 기록한다.
