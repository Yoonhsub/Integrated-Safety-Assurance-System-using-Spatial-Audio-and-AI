> 작성 목적: 지금까지 이 서비스의 **API 키 발급·GCP VM 개설·배포·운영을 혼자 담당하던 사람이 팀에서 빠지면서**, 남는 팀원이 서비스를 끊김 없이 안정적으로 운영하기 위해 알아야 할 모든 것을 정리한 문서입니다.

## 0. 서비스 한눈에 보기

MOBI는 청주 시내버스 기반의 **교통약자 안내 음성 에이전트**(passenger app)입니다. 사용자가 말로 목적지를 말하면, 위치·버스 정보를 그라운딩해서 **자연어 응답 + 음성(TTS)**으로 안내합니다.

```
[사용자 브라우저] ── HTTPS ──> [Caddy :80/:443]
   (Flutter Web)                    │ reverse_proxy
                                    ▼
                          [FastAPI / uvicorn :8000]  ← systemd: mobi-backend.service
                                    │
                    ┌───────────────┼────────────────────┐
                    ▼               ▼                     ▼
              OpenAI API       Gemini API          청주 버스 공공데이터 API
          (TTS/NLU/STT/그라운딩)  (폴백)              (정류장·노선·도착정보)
```

- **프론트엔드**: Flutter Web (passenger_app). FastAPI가 `/app/` 경로로 정적 서빙.
- **백엔드**: Python FastAPI(uvicorn). 음성·NLU·위치 그라운딩·버스 정보 오케스트레이션.
- **AI**: 기본은 OpenAI(현재 운영 기본 음성/NLU 경로), 폴백으로 Gemini. 토글로 전환 가능.
- **인프라**: GCP VM 1대 + Caddy 리버스 프록시 + nip.io 도메인.

핵심 소스 트리(레포 기준): `mobi-smart-transport-ai/` 하위에 `backend/`, `apps/passenger_app/` 등이 있습니다(레포 루트가 아니라 한 단계 안쪽 — §6 참고).

## 3. 접속 방법 (SSH / IAM)

### 3-1. gcloud SSH (IAP 터널 경유)
```bash
gcloud compute ssh instance-20260330-105638 \
  --zone=us-central1-b --project=gen-lang-client-0309873247 \
  --tunnel-through-iap
```
- 파일 복사(scp)도 같은 플래그:
```bash
gcloud compute scp --zone=us-central1-b --project=gen-lang-client-0309873247 \
  --tunnel-through-iap <로컬파일> instance-20260330-105638:/tmp/<이름>
```

### 3-2. SSH에 필요한 IAM 역할 (중요 — 권한 모자라면 SSH 자체가 실패)
이 프로젝트는 **OS Login 비활성** 상태라 SSH 키가 인스턴스 메타데이터로 푸시됩니다. 그래서 단순 `compute.viewer`나 `osLogin`만으론 **SSH가 안 됩니다.** 팀원에게 SSH를 주려면 프로젝트 레벨에서:
- `roles/compute.instanceAdmin.v1` (프로젝트)
- `roles/iam.serviceAccountUser` (컴퓨트 SA `132324468070-compute@developer.gserviceaccount.com`)

두 역할이 **둘 다** 있어야 합니다. (과거에 `get + osLogin`만 줬다가 SSH 실패한 이력 있음.)
> `instanceAdmin.v1`은 권한 범위가 넓으니, 시연/운영 안정화가 끝나면 회수 권장.

### 3-3. 권한 부여 명령 예시
```bash
gcloud projects add-iam-policy-binding gen-lang-client-0309873247 \
  --member="user:후임이메일" --role="roles/compute.instanceAdmin.v1"
gcloud iam service-accounts add-iam-policy-binding \
  132324468070-compute@developer.gserviceaccount.com \
  --member="user:후임이메일" --role="roles/iam.serviceAccountUser" \
  --project=gen-lang-client-0309873247
```

### 3-4. ⚠️ `sst70` 홈 디렉터리 접근 주의
- SSH로 들어간 내 계정은 `sst70` 홈(`/home/sst70`)에 **`cd` 못 합니다(퍼미션 700).**
- 라이브 트리 파일을 다룰 땐 **항상 `sudo`(또는 `sudo -u sst70`) + 절대경로**를 쓰세요. 상대경로/`cd`로 들어가려 하면 막힙니다.

## 5. 환경변수 / `.env` (운영 핵심 키만)

`.env`는 **gitignore라 레포에 없습니다**. VM 라이브 트리에만 있습니다. 전체 변수 목록은 레포의 `.env.example` 참고. 운영에 직접 영향 주는 핵심:

```ini
# === LLM (현재 운영 기본 = OpenAI) ===
OPENAI_API_KEY=sk-proj-...          # ★ 떠나는 담당자 개인 키 → 팀 키로 교체 대상
OPENAI_TTS_MODEL=gpt-realtime-2     # 프리미엄 Realtime 음성
OPENAI_TTS_VOICE=marin
OPENAI_REALTIME_MODEL=gpt-realtime-2
OPENAI_STT_MODEL=gpt-4o-mini-transcribe
LIVE_AUDIO_PROVIDER=openai          # Live 음성모드 제공자
# OPENAI_NLU_MODEL=gpt-4.1-mini     # 코드 기본값(미설정 시), NLU 모델
# OPENAI_GROUNDING_MODEL=gpt-4.1    # 코드 기본값(미설정 시), 위치 그라운딩(web search)

GEMINI_API_KEY=...                  # 폴백용. 현재 결제 이슈로 죽어 있을 수 있음(§10)
GEMINI_FLASH_MODEL=gemini-2.5-flash
GEMINI_PRO_MODEL=gemini-2.5-pro

# === 데이터 소스 ===
PUBLIC_DATA_USE_MOCK=true           # 공공데이터 mock 사용(운영 안정 위해 기본 mock)
FIREBASE_USE_MOCK / FCM_USE_MOCK 등 # mock fallback 위주
```

- **모델 ID 주의**: `gpt-realtime-2` 는 유효, `gpt-realtime-2.0`은 `model_not_found`. 풀사이즈 `gpt-audio`/`gpt-realtime`은 `/audio/speech`에서 404(반드시 chat/websocket 경로).
- `PUBLIC_DATA_USE_MOCK=true`로 두면 공공데이터 키 만료/장애와 무관하게 데모가 돕니다. 실데이터를 쓰려면 `false` + 유효한 `PUBLIC_DATA_API_KEY` 필요.

## 6. 코드 배포 방법

> ⚠️ **로컬 작업 디렉터리(`/Users/.../Downloads/mobi-smart-transport-ai 4`)는 git 저장소가 아니며 main보다 오래된 스냅샷입니다. 절대 여기서 빌드·배포하지 마세요.** 과거에 stale 로컬에서 프론트를 빌드·배포해 라이브의 최신 작업을 통째로 되돌린 사고가 있었습니다. **항상 `main`을 새로 clone 해서 거기서 작업하세요.**

배포 전 항상 백업:
```bash
L=/home/sst70/mobi-deploy/mobi-smart-transport-ai
sudo tar czf /home/sst70/mobi-backups/predeploy-$(date +%Y%m%d-%H%M%S).tar.gz \
  --exclude='.venv' -C "$(dirname "$L")" "$(basename "$L")"
```

### 6-1. 백엔드 배포 (fresh clone → rsync)
VM에서:
```bash
# 1) 최신 main을 새로 clone (레포 PUBLIC → 인증 불필요)
cd /tmp && rm -rf mobi-deploy-stage
git clone --branch main --depth 1 \
  https://github.com/Yoonhsub/Integrated-Safety-Assurance-System-using-Spatial-Audio-and-AI mobi-deploy-stage

# 2) rsync — ★ 소스 경로는 반드시 .../mobi-smart-transport-ai/ (레포 한 단계 안쪽!)
#    ★ build/ 는 제외(프론트 실체라 지우면 화면 사라짐), .env/.venv/.git 제외
L=/home/sst70/mobi-deploy/mobi-smart-transport-ai
STAGE=/tmp/mobi-deploy-stage/mobi-smart-transport-ai
sudo rsync -rin --checksum \
  --exclude={.env,.venv,.git,__pycache__,*.pyc,.pytest_cache,apps/passenger_app/build/} \
  "$STAGE/" "$L/"      # 먼저 드라이런(-rin)으로 변경파일 + 에러 확인 ★

# 위가 의도한 변경만 보이면 실제 적용:
sudo rsync -a --delete --chown=sst70:sst70 \
  --exclude={.env,.venv,.git,__pycache__,*.pyc,.pytest_cache,apps/passenger_app/build/} \
  "$STAGE/" "$L/"

# 3) 의존성 갱신(필요 시) + 재시작 + 스모크
sudo -u sst70 "$L/.venv/bin/pip" install -r "$L/backend/api/requirements.txt"
sudo systemctl restart mobi-backend
curl -s -o /dev/null -w '%{http_code}\n' http://127.0.0.1:8000/app/   # 200 확인
```

**⚠️ rsync 함정 3가지(과거 사고 원인)**:
1. 소스 경로를 레포 루트로 잘못 주면 `--delete`가 **전체를 삭제**. 반드시 `$STAGE/mobi-smart-transport-ai/`.
2. `apps/passenger_app/build/` 를 exclude 안 하면 **프론트가 통째로 삭제**(build는 gitignore라 main에 없음).
3. 항상 **드라이런 `-rin --checksum` + 에러 표시**로 먼저 확인. 에러를 숨기면 빈 결과를 "차이 없음"으로 오판함.

**실패 시 롤백**: 백업 tar를 overlay 복원. `.venv`는 백업에서 제외했으니 `rm` 금지(트리를 통째로 지우지 말고 백업을 덮어쓰기).

### 6-2. 단일 파일 핫패치(빠른 수정)
```bash
gcloud compute scp ... <로컬파일> instance-...:/tmp/<이름>     # (§3-1 플래그)
# VM에서:
L=/home/sst70/mobi-deploy/mobi-smart-transport-ai
sudo cp -a "$L/<대상경로>" /home/sst70/mobi-backups/           # 백업

sudo -u sst70 "$L/.venv/bin/python" -m py_compile "$L/<대상경로>"   # 절대경로! 문법검증
sudo systemctl restart mobi-backend
```

### 6-3. 프론트 배포 (로컬에서 빌드 → build/web만 올림)
VM엔 flutter가 없으니 **반드시 로컬(또는 빌드 가능 머신)에서 빌드**합니다. 최신 main clone 안의 `apps/passenger_app`에서:
```bash
flutter pub get
flutter build web --release --base-href /app/ \
  --dart-define=MOBI_API_BASE_URL=https://mobi.35.232.72.197.nip.io \

# ★ --base-href /app/ 없으면 배포 후 화면이 완전 백지(§10) — 빌드 직후 반드시 확인:

# 재시작 불필요. 검증:
```

## 10. 트러블슈팅 (실제로 겪은 장애 위주)

### 🔴 음성·채팅이 "앵무새"처럼 같은 말만 반복 / NLU 죽음
- **원인 1 (가장 흔함): Gemini 키 결제 문제.** 과거 두 번 발생 — ① `403 PERMISSION_DENIED: dunning`(결제 정지), ② `429 RESOURCE_EXHAUSTED: prepayment credits depleted`(선불 크레딧 소진). 증상: `/agent/converse`가 `usedGemini:false`, trace `reasoningSource:"keyword"`, `ttsMode:"LOCAL_TTS"`.
- **현재 구조에선**: 운영 기본이 OpenAI라 **OpenAI 키가 살아 있으면 Gemini가 죽어도 괜찮습니다.** 즉 이 증상이 보이면 **OpenAI 키/결제부터 확인.** OpenAI도 죽었으면 둘 다 막힌 것.
- **확인**: 
  ```bash
  KEY=$(sudo grep -E '^OPENAI_API_KEY=' /home/sst70/mobi-deploy/mobi-smart-transport-ai/.env | cut -d= -f2-)
  curl -s -o /dev/null -w '%{http_code}\n' -H "Authorization: Bearer $KEY" https://api.openai.com/v1/models
  ```
  401/429면 키/결제 문제 → §1-2 절차로 새 키 교체.
- **해결**: 해당 제공자 결제 복구(크레딧 충전/유료 전환) 또는 키 교체 → restart 불필요(키 그대로면) / 키 바꿨으면 `restart`.

### 🔴 배포 후 화면이 완전 백지(흰 화면)
- **원인**: 프론트를 `--base-href /app/` 없이 빌드함. 그러면 브라우저가 `/flutter_bootstrap.js`를 루트에서 찾아 404 → 백지. (API 스모크는 통과해서 놓치기 쉬움.)
- **해결**: `--base-href /app/`로 재빌드(§6-3). `grep '<base href' build/web/index.html`이 `/app/`인지 확인 후 배포.

### 🔴 배포 후 "백엔드 연결 실패: Load failed"
- **원인**: 프론트가 `MOBI_API_BASE_URL` 없이 빌드돼 `http://localhost:8000`을 호출. (현재 코드엔 `resolveApiBaseUrl()`가 web에서 `Uri.base.origin`으로 폴백하게 돼 있어 대부분 괜찮지만, 그게 빠졌다면) `--dart-define=MOBI_API_BASE_URL=https://mobi.35.232.72.197.nip.io`로 빌드.

### 🟡 rsync 배포가 프론트/전체를 지움
- §6-1의 "함정 3가지" 참고. 소스 경로(`/mobi-smart-transport-ai/`)·`build/` exclude·드라이런 필수.

### 🟡 Live STT(실시간 전사) 품질/에러
- OpenAI 전사 세션은 입력 샘플레이트 ≥24kHz 요구인데 브라우저 마이크는 16kHz로 보냄 → 백엔드가 16k→24k 업샘플 후 전달하도록 처리돼 있음(`v3_openai_live_audio_service.py`). 전사가 안 되면 이 경로/모델(`OPENAI_STT_MODEL`)부터 의심. 실측은 실제 마이크로 "말하기→전사→응답→음성" 한 바퀴 확인.

### 🟡 SSH가 안 됨
- IAM 역할 부족(§3-2). `instanceAdmin.v1` + `serviceAccountUser` 둘 다 필요.

### 🟡 VM 재부팅 후 URL 접속 불가
- 공인 IP가 ephemeral이라 바뀌었을 수 있음 → nip.io URL의 IP도 달라짐. **고정 IP 할당 권장**(§1-3). Caddy/백엔드 서비스가 `enabled`인지(`systemctl is-enabled`)도 확인.
