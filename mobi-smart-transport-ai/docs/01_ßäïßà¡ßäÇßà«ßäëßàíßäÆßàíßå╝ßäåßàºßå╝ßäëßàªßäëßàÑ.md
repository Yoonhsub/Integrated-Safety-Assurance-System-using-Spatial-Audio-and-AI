# 모비 프로젝트 요구사항명세서(SRS)

**프로젝트명:** 시각 장애인과 노약자를 위한 스마트 교통 AI 에이전트 시스템  
**문서 버전:** v1.0  
**작성일:** 2026-05-05  
**작성 기준:** 원본 캡스톤디자인 계획 문서 `모비 (2).docx` 및 수정된 4월 개인별 구현 범위 `mobi_april_personal_implementation_plan_revised.md`  

---

## 1. 문서 개요

### 1.1 목적
본 문서는 모비 프로젝트의 전체 프로그램에 대한 요구사항을 전문적인 소프트웨어 요구사항명세서 형태로 정리한다. 원본 문서가 프로젝트 기획과 월별/주차별 역할 분담 중심이라면, 본 문서는 실제 개발과 검증에 사용할 수 있도록 기능 요구사항, 비기능 요구사항, 데이터 요구사항, API 요구사항, 외부 인터페이스, 테스트 기준, 역할 추적성을 명시한다.

### 1.2 적용 범위
본 SRS는 승객용 Flutter 앱, 기사용 앱/화면, FastAPI 백엔드, Firebase Realtime Database, FCM 알림, 공공데이터 API 연동, BLE 비콘/센서 모듈, 2학기 AI 비전 준비 범위를 포함한다. 4월 MVP에서는 외부 헤드트래킹 센서 기반 구현을 제외하고, 스마트폰 방향 센서와 BLE/RSSI 기반 기초 기능까지만 요구한다.

### 1.3 우선순위 표기
- **M(Must):** 4월 MVP 또는 1학기 중간 산출물에 반드시 필요한 요구사항.
- **S(Should):** 가능하면 구현해야 하며, 통합 품질과 발표 완성도를 높이는 요구사항.
- **C(Could):** 여건이 되면 구현하는 선택 요구사항.
- **F(Future):** 2학기 또는 하드웨어 확보 이후 구현할 확장 요구사항.

### 1.4 참고 문서
- `모비 (2).docx`: 원본 캡스톤디자인 계획서, 프로젝트 주제, 주요 기능, 필요 기술, 월별/주차별 계획.
- `mobi_april_personal_implementation_plan_revised.md`: 4월 말까지 개인별 구현 범위 수정안. 헤드트래킹 제외, 공공데이터 API 김도성 이관, 김도성 테스트 계획 제외 사항 반영.

### 1.5 용어 정의
| 용어 | 정의 |
| --- | --- |
| STT | Speech-To-Text. 사용자의 음성을 텍스트로 변환하는 기능. |
| TTS | Text-To-Speech. 텍스트 안내 문구를 음성으로 출력하는 기능. |
| FCM | Firebase Cloud Messaging. 앱으로 푸시 알림을 보내는 Firebase 서비스. |
| 지오펜싱 | 위치 좌표가 특정 안전구역/위험구역 내부 또는 외부에 있는지 판별하는 기술. |
| BLE | Bluetooth Low Energy. 비콘 신호 수신 및 근거리 위치 추정에 사용하는 저전력 블루투스. |
| RSSI | Received Signal Strength Indicator. 수신 신호 세기. 비콘과의 대략적 거리 추정에 사용한다. |
| 저상버스 | 승하차 접근성이 높은 버스. 교통약자에게 우선 안내할 수 있다. |
| Ride Request | 승객이 특정 정류장/노선/버스에 대해 탑승 의사를 표시한 요청 데이터. |
| 4월 MVP | 4월 말까지 구현 가능한 최소 기능 묶음. 헤드트래킹 센서 기반 기능은 제외한다. |

## 2. 전체 시스템 설명

### 2.1 제품 관점
모비 시스템은 시각장애인과 고령자가 버스 정류장에서 안전하게 대기하고, 목적 버스를 확인하며, 버스 기사에게 탑승 요청을 전달할 수 있도록 돕는 교통약자 보조 시스템이다. 사용자는 음성 중심 앱을 통해 버스 정보와 안전 경고를 받고, 기사는 기사용 앱을 통해 교통약자 승객의 대기 사실을 사전에 확인한다.

### 2.2 주요 사용자
| ID | 사용자 | 설명 |
| --- | --- | --- |
| A-01 | 승객 사용자 | 시각장애인, 고령자, 일시적으로 이동 보조가 필요한 사용자. 음성 중심 인터페이스와 단순 UI를 사용한다. |
| A-02 | 버스 기사 | 기사용 앱 또는 기사 화면을 통해 교통약자 승객의 탑승 요청을 확인하고 상태를 갱신한다. |
| A-03 | 개발/운영자 | Firebase, FastAPI 서버, 공공데이터 API 키, 테스트 데이터, 로그를 관리한다. |
| A-04 | 발표/평가자 | 캡스톤 중간/최종 발표에서 기능 시연, 코드 기여도, 문서화 상태를 검토한다. |

### 2.3 시스템 구성요소
| ID | 구성요소 | 책임 |
| --- | --- | --- |
| C-01 | 승객용 Flutter 앱 | 사용자 UI, STT/TTS, 접근성, GPS 권한, 안전 상태/버스 정보 표시, 탑승 요청 생성 |
| C-02 | 기사용 Flutter 앱/화면 | 탑승 요청 목록, FCM 수신, 요청 상태 변경, 승객 대기 정보 표시 |
| C-03 | FastAPI 백엔드 | 지오펜싱 판별, Firebase Admin 연동, FCM 발송, ride request API, 버스 정보 표준 응답 인터페이스 |
| C-04 | Firebase Realtime Database | 사용자/기사/정류장/지오펜스/버스도착/탑승요청/토큰/알림 상태 저장 |
| C-05 | Firebase Cloud Messaging | 위험 경고, 탑승 요청, 기사 확인/도착 등 푸시 알림 전달 |
| C-06 | 공공데이터 API 연동 모듈 | 정류장별 도착 정보, 노선/차량 위치, 저상버스 여부, 가능 시 혼잡도 조회 및 표준화 |
| C-07 | BLE/센서 모듈 | 스마트폰 방향 센서, BLE 비콘 스캔, RSSI 기반 거리 등급 산출 |
| C-08 | AI 비전 준비 모듈 | 2학기 카메라 객체 탐지 기능을 위한 데이터 수집 계획, 샘플 데이터, 모델 리서치 |

### 2.4 운영 환경
- **모바일 앱:** Flutter 기반 Android 우선 구현, iOS 확장 가능 구조.
- **백엔드:** Python FastAPI. 개발 단계에서는 로컬 서버 또는 클라우드 서버 사용 가능.
- **데이터 저장소:** Firebase Realtime Database.
- **알림:** Firebase Cloud Messaging.
- **외부 데이터:** 공공데이터포털 또는 지자체 버스 도착/위치 API.
- **센서/하드웨어:** 스마트폰 GPS, 마이크, 스피커, 진동, 블루투스, 방향 센서, BLE 비콘. 외부 헤드트래킹 센서는 4월 범위에서 제외.

### 2.5 핵심 제약사항
- 위치 정보는 GPS 오차가 있으므로 위험 판정은 완충 구역과 반복 판정을 고려해야 한다.
- 공공데이터 API는 지역, 인증키, 응답 형식, 호출 제한에 따라 기능이 제한될 수 있다.
- FCM은 네트워크와 OS 정책에 따라 즉시 도착을 보장하지 않는다.
- BLE RSSI는 환경 영향을 크게 받으므로 4월에는 정밀 거리 대신 거리 등급 중심으로 구현한다.
- 헤드트래킹 센서가 아직 없으므로 해당 기능은 4월 MVP에서 제외한다.

## 4. 기능 요구사항

아래 기능 요구사항은 개발 및 검증을 위해 고유 ID, 우선순위, 담당자, 입력, 출력, 인수 기준을 포함한다.

### FR-001. 앱 실행 및 초기 권한 요청
- **우선순위:** M
- **담당:** 윤현섭
- **요구사항 설명:** 승객 앱은 최초 실행 시 위치, 마이크, 알림, 블루투스 사용 목적을 명확히 설명하고 필요한 권한을 요청해야 한다.
- **입력:** 앱 최초 실행, OS 권한 상태
- **출력:** 권한 요청 화면, 거부 시 대체 안내
- **인수 기준:** 권한 허용/거부/부분 허용 각각에서 앱이 중단되지 않고 다음 행동을 안내한다.

### FR-002. 승객용 홈 화면
- **우선순위:** M
- **담당:** 윤현섭
- **요구사항 설명:** 승객용 앱은 앱 실행 직후 핵심 기능에 접근 가능한 단순 홈 화면을 제공해야 한다. 화면은 초대형 버튼과 큰 글씨 중심이어야 한다.
- **입력:** 앱 실행 이벤트
- **출력:** 음성 입력, 버스 정보, 안전 상태, 탑승 요청 버튼
- **인수 기준:** 시각적 복잡도가 낮고 주요 버튼이 한 화면에서 접근 가능하다.

### FR-003. STT 기반 음성 입력
- **우선순위:** M
- **담당:** 윤현섭
- **요구사항 설명:** 사용자는 음성으로 목적지, 버스 번호, 도착 정보 질문, 탑승 요청을 입력할 수 있어야 한다.
- **입력:** 마이크 입력, 사용자의 음성
- **출력:** 인식된 텍스트, 인식 실패 메시지
- **인수 기준:** 예시 문장 “502번 버스 언제 와?”를 텍스트로 표시하고 재시도 버튼을 제공한다.

### FR-004. TTS 기반 음성 안내
- **우선순위:** M
- **담당:** 윤현섭
- **요구사항 설명:** 시스템은 버스 도착, 위험 경고, 탑승 요청 상태를 음성으로 안내해야 한다.
- **입력:** 안내 문구, 안전 상태, 버스 정보, 요청 상태
- **출력:** 스피커 음성 출력, 화면 텍스트 병행 표시
- **인수 기준:** TTS 실패 시에도 화면 텍스트가 유지되고 앱이 멈추지 않는다.

### FR-005. 접근성 라벨 및 TalkBack 대응
- **우선순위:** M
- **담당:** 윤현섭
- **요구사항 설명:** 모든 주요 버튼과 상태 표시는 스크린리더가 의미를 읽을 수 있도록 접근성 라벨을 가져야 한다.
- **입력:** 화면 위젯
- **출력:** Semantics/TalkBack 라벨
- **인수 기준:** 주요 버튼을 스크린리더로 탐색했을 때 기능이 명확히 읽힌다.

### FR-006. 버스 정보 화면 렌더링
- **우선순위:** M
- **담당:** 윤현섭
- **요구사항 설명:** 승객 앱은 백엔드 또는 mock JSON에서 받은 버스 도착 정보를 카드 형태로 표시해야 한다.
- **입력:** 버스 정보 JSON
- **출력:** 노선번호, 도착예정, 남은 정류장, 저상버스 여부, 혼잡도
- **인수 기준:** API가 없을 때 mock 데이터로도 동일한 UI를 시연할 수 있다.

### FR-007. 안전 상태 경고 UI
- **우선순위:** M
- **담당:** 윤현섭
- **요구사항 설명:** 앱은 지오펜싱 결과에 따라 안전/주의/위험 상태를 명확하게 표시해야 한다.
- **입력:** 지오펜싱 판별 결과
- **출력:** 상태 배너, 음성 안내, 진동 트리거 요청
- **인수 기준:** SAFE/WARNING/DANGER/OUT_OF_AREA 상태가 서로 구분되어 표시된다.

### FR-008. 기사용 기본 화면
- **우선순위:** S
- **담당:** 윤현섭
- **요구사항 설명:** 기사용 화면은 승객 탑승 요청 목록과 요청 상태를 표시할 수 있어야 한다.
- **입력:** rideRequests 데이터
- **출력:** 요청 카드, 승객 유형, 정류장, 노선, 상태
- **인수 기준:** 최소 1건의 mock 요청이 기사 화면에 표시된다.

### FR-020. 스마트폰 방향 센서 수집
- **우선순위:** M
- **담당:** 안준환
- **요구사항 설명:** 센서 모듈은 스마트폰의 나침반/방향 값을 읽고 패키지 내부 예제 또는 로그에 표시해야 한다. 외부 헤드트래킹 센서는 4월 구현 범위에서 제외한다. 실제 사용자/기사 앱 화면 반영은 윤현섭 담당 앱 영역에서 수행한다.
- **입력:** 스마트폰 방향 센서
- **출력:** headingDegrees, accuracy, updatedAt
- **인수 기준:** 센서 사용 가능 기기에서 headingDegrees 값이 주기적으로 갱신된다.

### FR-021. BLE 비콘 스캔
- **우선순위:** M
- **담당:** 안준환
- **요구사항 설명:** 시스템은 주변 BLE 기기를 스캔하고 프로젝트에서 지정한 비콘을 식별해야 한다.
- **입력:** BLE 광고 패킷, 권한 상태
- **출력:** beaconId, rssi, estimatedDistanceMeters, signalLevel, lastDetectedAt

| rideRequests/{requestId} | 승객 탑승 요청 | userId, stopId, routeId, busNo, targetDriverId, status, createdAt, updatedAt |
| fcmTokens/{ownerType}/{ownerId} | FCM 토큰 저장소 | token, platform, updatedAt |
주의: 4월 MVP의 공식 Firebase RTDB 최상위 경로는 위 표의 `/users`, `/drivers`, `/busStops`, `/geofences`, `/busArrivals`, `/rideRequests`, `/fcmTokens`, `/systemLogs`로 한정한다.

- 기존 설계 초안의 `alerts/{alertId}`는 4월 공식 RTDB 경로가 아니다. 위험/매칭/상태 알림 기록은 필요 시 `/systemLogs`에 기록하거나 향후 확장 항목으로 별도 설계한다.
- 기존 설계 초안의 `beaconSignals/{deviceId}`는 4월 공식 RTDB 경로가 아니다. 비콘 신호는 `packages/mobile_sensors` 내부 디버그 로그 또는 향후 확장 항목으로 다룬다.

주의: `busArrivals`의 app-facing API 응답에는 `stopName`과 `source`를 포함하지 않는다. 공공데이터 원천/제공처가 필요하면 Firebase 내부 캐시 메타데이터 또는 로그에만 보관하고 Flutter 응답 계약에는 노출하지 않는다.

### 6.2 핵심 상태값

#### 6.2.1 지오펜싱 상태
```txt
SAFE: 안전구역 내부
WARNING: 주의구역 또는 위험구역 접근
DANGER: 위험구역 진입
OUT_OF_AREA: 정류장 관리 범위 밖
UNKNOWN: 좌표/정류장/구역 정보 부족
```

#### 6.2.2 탑승 요청 상태
```txt
WAITING: 승객 요청 생성
NOTIFIED: 기사에게 알림 발송
ACCEPTED: 기사가 확인
ARRIVED: 버스 도착 또는 도착 처리
COMPLETED: 탑승 완료
CANCELLED: 승객/기사/시스템에 의해 취소
```

#### 6.2.3 비콘 거리 등급
```txt
VERY_CLOSE: 비콘이 매우 가까움
CLOSE: 가까움
MEDIUM: 중간 거리
FAR: 멀어짐
LOST: 신호 소실 또는 일정 시간 미감지
```

## 7. 외부 인터페이스 요구사항

### 7.1 사용자 인터페이스

- 모든 주요 버튼은 스크린리더 라벨을 가져야 한다.
- 위험 상태는 색상만으로 구분하지 말고 텍스트, 음성, 진동을 함께 사용해야 한다.
- 기사 화면은 요청 목록을 카드 형태로 표시하고 상태 변경 버튼을 제공해야 한다.

### 7.2 하드웨어/센서 인터페이스

- 마이크: STT 입력에 사용한다.
- 스피커: TTS 안내에 사용한다.
- 진동: 위험 경고와 근접 안내에 사용한다.
- Bluetooth: BLE 비콘 스캔에 사용한다.
- 스마트폰 방향 센서: 4월 범위에서 사용 가능하다.
- 외부 헤드트래킹 센서: 4월 범위에서 제외하며, 향후 확장 모듈로 남긴다.

### 7.4 공공데이터 API 인터페이스
공공데이터 API 연동은 김도성이 담당한다. 원본 응답을 앱에 직접 노출하지 않고 표준 JSON으로 변환해야 하며, API 장애 또는 데이터 없음 상황에서 mock 데이터 또는 명확한 에러 응답을 제공해야 한다.

## 8. 오류 및 예외 처리 요구사항
- 위치 권한 거부: 위치 기반 기능을 사용할 수 없음을 안내하고 수동 정류장 선택 또는 mock 모드를 제공한다.
- 마이크 권한 거부: 음성 입력 대신 텍스트/버튼 입력 대안을 제공한다.
- 알림 권한 거부: FCM 대신 앱 내 상태 표시를 유지한다.
- 공공데이터 API 실패: 사용자에게 “현재 버스 정보를 불러올 수 없습니다”를 안내하고 재시도 버튼을 제공한다.
- GPS 오차: 즉시 위험 확정 대신 WARNING 상태를 거쳐 DANGER로 전환한다.
- BLE 신호 소실: LOST 상태로 표시하고 사용자를 혼란스럽게 하는 급격한 안내를 피한다.
- Firebase 연결 실패: 로컬 상태 표시와 재시도 로직을 제공한다.

## 9. 보안 및 개인정보 요구사항
- API 키, Firebase 서비스 계정, FCM 관련 비밀값은 GitHub에 커밋하지 않는다.
- 위치 정보는 기능 수행에 필요한 최소 범위에서만 저장한다.
- 테스트 데이터에는 실제 이름, 실제 장애 정보, 실제 민감 위치를 넣지 않는다.
- FCM 토큰은 사용자/기사 식별자와 연결되므로 외부에 노출하지 않는다.
- 운영 전 단계에서는 Firebase DB 규칙을 최소 권한 원칙에 맞게 설정해야 한다.
- 앱 로그에는 API 키, 토큰, 민감 위치 원본 전체를 그대로 출력하지 않는다.
