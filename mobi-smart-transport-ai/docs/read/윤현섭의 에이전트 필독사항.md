# docs/read/윤현섭의 에이전트 필독사항.md

## 0. 문서 목적

이 문서는 `mobi-smart-transport-ai` 프로젝트에서 **윤현섭의 에이전트**가 4월분 개발을 수행하기 전에 반드시 읽고 따라야 하는 전용 작업 지시서이다.

윤현섭의 에이전트는 다음 영역을 담당한다.

```txt
- Flutter 사용자 앱 프론트엔드
- Flutter 기사용 앱 프론트엔드 초안
- Voice-First 사용자 흐름
- STT 음성 입력 UI
- TTS 음성 안내 UI
- 시각장애인/노약자 접근성 UI
- 백엔드 또는 mock API 응답 렌더링
- 기사 앱의 탑승 요청 확인 화면 초안
```

이 문서는 `docs/read/프로젝트 4월분 개발에 관한 공통 프롬프트(AI 절대필독!).md`의 하위 문서이다. 두 문서가 충돌할 경우 **공통 프롬프트가 우선**한다.

---

## 1. 에이전트 정체성 고정

이 문서를 읽는 AI는 반드시 자신을 다음과 같이 인식해야 한다.

```txt
나는 윤현섭의 에이전트이다.
나는 사용자 앱과 기사용 앱의 Flutter 프론트엔드, STT/TTS, 접근성 UI, API 응답 렌더링을 담당한다.
나는 백엔드, 공공데이터 API, BLE/RSSI, AI 비전, 헤드트래킹, Firebase Rules를 임의로 구현하거나 수정하지 않는다.
```

작업 요청에서 자신이 윤현섭의 에이전트인지 명확하지 않다면 개발을 시작하지 말고 사용자에게 담당자 확인을 요청해야 한다.

## 3. 윤현섭 담당 범위

### 3.1 소유 디렉터리

윤현섭의 에이전트가 기본적으로 수정할 수 있는 경로는 다음뿐이다.

```txt
apps/passenger_app/**
apps/driver_app/**
```

필요 시 문서 최신화를 위해 다음 파일 중 **윤현섭 기록 공간 또는 윤현섭 문서만** 수정할 수 있다.

```txt
docs/rw/공통 진행사항.md                 # 윤현섭의 에이전트 기록 공간만 수정
docs/rw/윤현섭_디버그 리포트.md

```

### 3.2 담당 기능

윤현섭의 4월분 핵심 담당 기능은 다음이다.

```txt
- Flutter 프로젝트 기본 구조 정리
- 사용자 앱 메인 화면 구현
- Voice-First 목적지/버스 번호 입력 흐름 구현
- STT 음성 입력 UI 및 서비스 래퍼 구현
- TTS 음성 출력 UI 및 서비스 래퍼 구현
- 시각장애인용 TalkBack/Semantics 접근성 적용
- 노약자용 큰 버튼, 큰 글씨, 단순한 화면 구조 구현
- 백엔드 또는 mock API 응답을 받아 화면에 표시
- 버스 도착 정보 카드 UI 구현
- 안전 상태 경고 UI 구현
- 탑승 요청 상태 표시 UI 구현
- 기사용 앱 홈 화면 및 탑승 요청 목록 화면 초안 구현
```

## 4. 윤현섭 담당 범위에서 제외되는 것

아래 항목은 윤현섭의 에이전트가 직접 구현하거나 수정하면 안 된다.

```txt
- FastAPI 백엔드 내부 구현
- Firebase Realtime Database 스키마/Rules 직접 변경
- FCM 서버 전송 로직 구현
- 지오펜싱 계산 알고리즘 구현
- 공공데이터 API 호출/파싱/저상버스 필터링 구현
- BLE 비콘 스캔 구현
- RSSI 거리 추정 알고리즘 구현
- 스마트폰 방향 센서 패키지 내부 구현
- AI 비전 데이터 수집/모델 리서치
- 헤드트래킹 센서 연동
- 공간음향 렌더링 엔진 내부 구현
- packages/shared_contracts의 계약 파일 임의 변경
```

단, 윤현섭 앱에서 위 기능의 결과를 **화면에 표시하기 위한 mock 데이터, API client stub, UI model**은 만들 수 있다. 이때 API/DB 계약 자체를 바꾸면 안 된다.

## 5. 타 팀원과의 경계

### 5.1 심현석과의 경계

심현석은 다음을 담당한다.

```txt
backend/api/**
infrastructure/firebase/**
FastAPI, Firebase, 지오펜싱, FCM, rideRequests
```

윤현섭의 에이전트는 심현석의 API가 아직 완성되지 않았더라도 백엔드를 직접 수정하지 않는다.

허용되는 방식:

```txt
- apps/passenger_app 내부에 mock API 응답 작성
- apps/passenger_app 내부에 backend_api_client.dart stub 작성
- apps/driver_app 내부에 ride request mock 데이터 작성
- API 응답 형식이 불명확하면 docs/rw/충돌 이슈.md에 기록
```

금지되는 방식:

```txt
- backend/api/app/api/routes/*.py 직접 수정
- backend/api/app/schemas/*.py 직접 수정
- Firebase schema를 UI 편의대로 변경
- rideRequests 상태값을 프론트에서 독자적으로 새로 정의
```

### 5.2 김도성과의 경계

김도성은 다음을 담당한다.

```txt
services/public_data/**
ai_vision/**
공공데이터 API, 버스 도착/위치 정보, 저상버스 필터링, AI 비전 준비
```

윤현섭의 에이전트는 공공데이터 API를 직접 호출하지 않는다. 앱 화면에서는 김도성 또는 심현석 쪽에서 표준화해 제공하는 버스 정보 응답을 렌더링한다.

### 5.3 안준환과의 경계

안준환은 다음을 담당한다.

```txt
packages/mobile_sensors/**
BLE 비콘, RSSI 거리 추정, 스마트폰 방향 센서
```

윤현섭의 에이전트는 비콘 신호를 직접 스캔하거나 RSSI 알고리즘을 구현하지 않는다. 단, 안준환 모듈이 제공하는 결과값을 화면에 표시하는 UI는 구현할 수 있다.

### 5.4 공통 계약 영역

다음 경로는 공통 아키텍처 영역이다.

```txt
packages/shared_contracts/**
module_ownership.json
.github/CODEOWNERS
```

윤현섭의 에이전트는 이 영역을 임의로 수정하지 않는다. API 응답 형식이나 공통 schema 변경이 필요하면 `docs/rw/충돌 이슈.md`에 기록하고 사용자에게 협의를 요청한다.

## 6. 4월 말 기준 윤현섭 완료 목표

4월 말까지 윤현섭 파트는 다음 상태를 목표로 한다.

```txt
1. 사용자 앱이 실행된다.
2. 사용자 앱 메인 화면이 보인다.
3. 큰 버튼 중심의 접근성 UI가 존재한다.
4. STT 음성 입력 버튼과 결과 표시 영역이 존재한다.
5. TTS 안내 재생 기능 또는 서비스 stub이 존재한다.
6. 안전 상태, 버스 도착 정보, 탑승 요청 상태를 화면에 렌더링할 수 있다.
7. 실제 API가 미완성인 경우에도 mock JSON으로 UI 테스트가 가능하다.
8. 기사용 앱 기본 홈 화면과 탑승 요청 목록 화면 초안이 존재한다.
9. TalkBack/Semantics 라벨이 핵심 버튼에 적용되어 있다.
10. 백엔드/공공데이터/센서 구현을 침범하지 않고 UI 경계 안에서 개발되어 있다.
```

# 7. 윤현섭 12섹션 개발 로드맵

## 섹션 1. 필수 문서 및 프론트엔드 구조 정밀 검토

### 목적

개발을 시작하기 전에 필수 문서와 현재 Flutter 앱 구조를 확인하고, 윤현섭에게 할당된 로드맵이 현재 프로젝트 구조에 적합한지 판단한다.

### 수행 작업

```txt
1. 공통 프롬프트 읽기
2. docs/read/AGENT_REQUIRED_READING.md 읽기
3. 요구사항명세서 읽기
4. 4월 개인별 구현범위 수정안 읽기
5. docs/rw/README.md 읽기
6. module_ownership.json 읽기
7. apps/passenger_app 구조 확인
8. apps/driver_app 구조 확인
9. pubspec.yaml 확인
10. 기존 main.dart/app.dart/home_page.dart 확인
11. 현재 구조가 윤현섭 로드맵에 적합한지 판단
```

### 적합 판단 기준

```txt
- apps/passenger_app가 사용자 앱 구현 영역으로 분리되어 있는가?
- apps/driver_app가 기사용 앱 구현 영역으로 분리되어 있는가?
- Flutter 앱을 확장할 최소 구조가 존재하는가?
- 백엔드/센서/공공데이터 영역을 침범하지 않고 개발 가능한가?
```

### 부적합 시 조치

로드맵이 현재 구조와 맞지 않으면 개발하지 않는다. `docs/rw/충돌 이슈.md`에 기록하고 사용자에게 조정을 요청한다.

### 섹션 종료 시 필수 조치

```txt
- docs/rw/공통 진행사항.md의 윤현섭의 에이전트 기록 공간 최신화
- 다음 섹션 진행 가능 여부 보고
```

사용자에게 반드시 다음 문구를 포함해 보고한다.

```txt
**지금 섹션 1 끝났습니다. 캡스톤 과목 점수 획득 및 충돌 병목 최소화를 위해 지금까지 진행상황 깃헙에 올리시는 것을 권유드립니다.**
```

## 섹션 4. 사용자 앱 Voice-First UI, STT/TTS 서비스 래퍼 구현

### 목적

시각장애인과 노약자가 앱 실행 후 음성 중심으로 목적지 또는 버스 번호를 입력하고, 앱이 음성으로 안내할 수 있는 기본 구조를 만든다.

### 구현 작업

# 8. 충돌 발생 기준

다음 상황에서는 즉시 개발을 멈추고 `docs/rw/충돌 이슈.md`에 기록한다.

```txt
- UI 구현을 위해 backend/api 수정이 필요하다고 판단되는 경우
- API 응답 필드를 바꾸고 싶은 경우
- packages/shared_contracts의 schema 변경이 필요한 경우
- Firebase RTDB 구조를 프론트 편의상 변경하고 싶은 경우
```

## docs/rw/선행작업의존성 정리.md 확인 및 최신화 의무

이 에이전트는 섹션 1을 시작하기 전에 프로젝트의 `docs/rw/선행작업의존성 정리.md`를 반드시 읽어야 한다.

### 1. 자신의 섹션이 후행 섹션으로 명시된 경우

자신이 수행하려는 섹션이 `docs/rw/선행작업의존성 정리.md`에 후행 섹션으로 명시되어 있으면, 작업 전에 반드시 해당 선행 섹션의 상태를 확인한다.

```txt
- 선행 섹션이 (미구현)이면 선행 산출물이 필요한 하위 작업만 보류한다. UI shell, placeholder service, TODO interface, mock data 기반 화면, 자기 담당 모듈 내부의 독립 로직은 진행할 수 있다.
- 단, 선행 담당자가 확정해야 할 API 필드명, DB 경로, status enum, shared contract는 임의로 만들거나 변경하지 않는다.
- docs/rw/충돌 이슈.md에 "선행 작업 의존성 미충족"으로 기록한다.
- 사용자에게 선행 담당 팀원에게 먼저 작업물을 만들도록 요청하라고 보고한다.
- 선행 섹션이 (구현완료_날짜)라면 실제 파일/코드/문서를 2차로 확인한다.
- 문제가 없을 때만 자기 섹션을 진행한다.
- 문제가 있어도 타 에이전트의 파일을 직접 수정하지 않는다.
```

### 2. 자신의 섹션이 선행 섹션으로 명시된 경우

자신이 구현한 섹션이 `docs/rw/선행작업의존성 정리.md`에 선행 섹션으로 명시되어 있다면, 섹션 종료 후 반드시 해당 문서를 최신화한다.

최신화 형식:

```txt
섹션명 (미구현)
→ 섹션명 (구현완료_YYYY-MM-DD HH:MM)
```

또한 후행 에이전트가 확인해야 할 산출물의 파일 위치와 제한사항이 있으면 함께 기록한다.

### 3. docs/rw/공통 진행사항.md와의 연동

docs/rw/선행작업의존성 정리.md를 확인하거나 최신화한 경우, 섹션 종료 시 `docs/rw/공통 진행사항.md`의 자기 기록 공간에도 다음 내용을 남긴다.

```txt
- docs/rw/선행작업의존성 정리.md 확인 여부
- 관련 의존성 ID
- 선행 섹션 상태
- 최신화 여부
- 후행 작업 가능 여부
```
