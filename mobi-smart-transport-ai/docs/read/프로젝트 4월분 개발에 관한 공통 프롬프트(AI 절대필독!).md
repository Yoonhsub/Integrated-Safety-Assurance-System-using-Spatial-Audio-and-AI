# 프로젝트 4월분 개발에 관한 공통 프롬프트(AI 절대필독!)

## 0. 이 문서의 목적

이 문서는 `mobi-smart-transport-ai` 프로젝트에서 4명의 팀원이 각자 생성형 AI 에이전트를 활용해 4월분 개발을 병렬 수행할 때, 역할 침범·중복 구현·API 계약 불일치·Git 병합 충돌·책임 소재 불명확성을 최소화하기 위한 **최상위 공통 작업 규칙**이다.

모든 AI 에이전트는 코드 작성, 파일 수정, 리팩토링, 테스트 추가, 문서 최신화 등 어떠한 작업을 시작하기 전에 반드시 이 문서를 읽고 따라야 한다.

이 문서는 다음을 강제한다.

```txt
1. 에이전트 신원 확인
2. 필수 문서 정독
3. 담당자별 개발 범위 준수
4. 타 팀원 영역 무단 수정 금지
5. 충돌 발생 시 즉시 기록 및 개발 중단
6. 섹션 단위 구현/검토/패치/보고
7. 공통 진행사항 및 디버그 리포트 최신화
8. GitHub 업로드 및 PR 단위 협업 유도
```

## 1. 최우선 원칙

### 1.1 에이전트 신원 확인 의무

AI 에이전트는 작업 시작 전에 반드시 자신이 어느 팀원의 에이전트인지 명확히 확인해야 한다.

허용되는 담당자명은 다음 4명뿐이다.

```txt
윤현섭의 에이전트
안준환의 에이전트
심현석의 에이전트
김도성의 에이전트
```

사용자 요청에 담당 팀원이 명시되지 않았거나, 현재 대화 맥락만으로 담당 팀원을 확정할 수 없다면 **절대 개발을 수행하지 않는다.**

이 경우 에이전트는 다음과 같이 응답해야 한다.

```txt
현재 제가 어느 팀원의 에이전트인지 명확히 지정되지 않았습니다.
이 프로젝트는 팀원별 담당 범위가 엄격히 분리되어 있으므로, 개발을 시작하기 전에 제가 윤현섭/안준환/심현석/김도성 중 누구의 에이전트인지 먼저 지정해 주세요.
```

### 1.2 담당자 외 영역 수정 금지

에이전트는 자신이 담당한 팀원의 역할 범위와 소유 디렉터리 안에서만 작업해야 한다.

다음 행위는 금지된다.

```txt
- 다른 팀원의 담당 폴더를 임의 수정하는 행위
- 다른 팀원의 API/DB/모듈 설계를 사전 협의 없이 변경하는 행위
- 자신의 구현 편의를 위해 공통 계약 파일을 독단적으로 바꾸는 행위
- 다른 팀원의 진행사항 기록 공간을 수정하는 행위
- 충돌 가능성을 알고도 docs/rw/충돌 이슈.md에 기록하지 않고 진행하는 행위
```

### 1.3 충돌 가능성 발견 시 개발 중단

자신의 작업을 계속하려면 다른 팀원의 담당 영역을 변경해야 하거나, API 계약·DB 스키마·공통 모델·폴더 소유권이 충돌할 가능성이 있다면 즉시 작업을 멈춘다.

그 후 반드시 프로젝트 루트의 다음 파일에 기록한다.

```txt
docs/rw/충돌 이슈.md
```

기록 후 사용자에게 굵은 글씨로 다음 취지의 메시지를 전달해야 한다.

```txt
**이 작업은 [팀원 이름] 담당 영역과 충돌 가능성이 있으므로 더 이상 개발을 진행할 수 없습니다. 반드시 [팀원 이름]에게 연락하여 이 이슈를 논의한 뒤 진행해야 합니다.**
```

### 2.1 일반 업무/역할 범위 우선순위

문서 간 내용이 충돌할 경우, 역할·범위·섹션 진행 판단은 다음 우선순위를 따른다. 단, API 필드명/응답 구조/enum, Firebase RTDB 경로/필드, 보안 규칙은 아래 2.1.1의 계약 영역별 우선순위가 이 일반 우선순위보다 우선한다.

```txt
1순위: 이 문서, docs/read/프로젝트 4월분 개발에 관한 공통 프롬프트(AI 절대필독!).md
2순위: 각 팀원별 에이전트 필독사항.md
3순위: docs/rw/선행작업의존성 정리.md
4순위: docs/02_4월_개인별_구현범위_수정안.md
5순위: docs/01_요구사항명세서.md
6순위: docs/rw/ARCHITECTURE.md, docs/rw/API_CONTRACTS.md, docs/rw/DATA_SCHEMA.md, docs/rw/MODULE_OWNERSHIP.md
7순위: docs/00_원본_모비_프로젝트_계획서.docx
```

단, `docs/00_원본_모비_프로젝트_계획서.docx`는 최신 충돌 수정본이어야 하며, 과거 계획과 현재 실행 계획이 다를 경우 현재 실행 계획을 우선한다.

중요: 위 표는 일반 업무/역할 범위 판단용이다. `docs/02_4월_개인별_구현범위_수정안.md`나 팀원별 문서의 예시가 shared JSON Schema, backend Pydantic schema, Firebase RTDB schema와 충돌하면 계약 영역별 우선순위가 우선한다.

### 2.2 4월 구현 기준 핵심 변경 사항

모든 에이전트는 아래 수정 기준을 반드시 따른다.

```txt
1. 안준환: 4월에는 헤드트래킹 센서 구현을 하지 않는다.
2. 안준환: 4월에는 BLE 비콘, RSSI 거리 추정, 스마트폰 방향 센서에 집중한다.
3. 심현석: 공공데이터 API 직접 연동을 담당하지 않는다.
4. 심현석: Firebase, FastAPI, 지오펜싱, FCM, 기사-승객 매칭 파이프라인을 담당한다.
5. 김도성: 공공데이터 API 기반 버스 정보 연동과 저상버스 필터링을 담당한다.
6. 김도성: 비콘이 몇 m 거리에서 잡히는지 테스트 계획을 담당하지 않는다.
7. 김도성: 공간음향이 방향을 제대로 주는지 테스트 계획을 담당하지 않는다.
8. 윤현섭: Flutter 사용자 앱, 기사 앱 UI, STT/TTS, 접근성, 백엔드 데이터 렌더링을 담당한다.
```

## 2.1.1 계약 영역별 우선순위 분리

문서 간 충돌이 발생했을 때는 모든 항목을 하나의 우선순위로 처리하지 않는다. 아래처럼 영역별 기준 문서를 분리한다.

### 역할/범위/섹션 진행 기준
1. `docs/read/프로젝트 4월분 개발에 관한 공통 프롬프트(AI 절대필독!).md`
2. 각 팀원별 `에이전트 필독사항.md`
3. `docs/02_4월_개인별_구현범위_수정안.md`
4. `docs/01_요구사항명세서.md`

### API 필드명/응답 구조/enum 기준
1. `packages/shared_contracts/api/*.schema.json`
2. `backend/api/app/schemas/*.py`
3. `docs/rw/API_CONTRACTS.md`
4. 각 팀원별 문서와 구현 범위 문서

### Firebase RTDB 경로/필드 기준
1. `infrastructure/firebase/realtime_database.schema.json`
2. `infrastructure/firebase/database.rules.json`
3. `docs/rw/DATA_SCHEMA.md`
4. `docs/01_요구사항명세서.md`

즉, `docs/02_4월_개인별_구현범위_수정안.md`는 역할과 범위 판단에는 높은 우선순위를 갖지만, API 필드명이나 DB 경로를 shared contract/schema보다 우선해서 바꾸는 근거가 될 수 없다.

## 3. 프로젝트 전체 개발 철학

이 저장소의 초기 목적은 완성 기능 구현이 아니라, 요구사항명세서 전체를 충실히 반영할 수 있는 **공통 프로젝트 아키텍처와 모듈 경계**를 세우는 것이다.

따라서 각 에이전트는 다음을 준수해야 한다.

```txt
- 세부 기능을 무리하게 완성하려고 하지 않는다.
- 자기 담당 기능의 확장 가능한 구조와 계약을 우선한다.
- 다른 팀원이 연결할 수 있는 인터페이스를 명확히 유지한다.
- 임시 구현은 TODO, mock, stub임을 명확히 표시한다.
- 향후 기능은 future_modules에 프레임만 두고 4월 구현 범위와 혼동하지 않는다.
- 공통 계약 변경은 충돌 가능성이 높으므로 단독 변경하지 않는다.
```

### 3.1 구현보다 중요한 것

4월 개발에서 가장 중요한 것은 다음이다.

```txt
1. 역할 경계 보존
2. 모듈 간 계약 안정성
3. Git 병합 가능성
4. 문서와 코드의 일관성
5. 섹션별 검토 기록
6. 팀원별 기여도 추적 가능성
```

## 5. 공통 12섹션 작업 규칙

모든 팀원의 에이전트는 4월분 개발을 12섹션으로 나누어 수행한다.

### 5.1 섹션 1: 사전 정밀 검토

섹션 1에서는 구현하지 않는다.

반드시 수행할 일:

```txt
1. 필수 문서 전체 확인
2. 자신의 팀원 전용 에이전트 필독사항 확인
3. module_ownership.json 확인
4. .github/CODEOWNERS 확인
5. 자신이 담당할 코드 디렉터리 구조 확인
6. 현재 프로젝트 구조와 자신의 12섹션 로드맵이 적합한지 판단
7. 충돌 가능성 여부 판단
```

판단 결과:

```txt
- 적합하면 섹션 2 진행 가능 보고
- 부적합하면 docs/rw/충돌 이슈.md에 기록하고 개발 중단
```

### 5.2 섹션 2, 4, 6, 8, 10: 직접 구현 단계

짝수 섹션은 직접 구현 단계이다.

규칙:

```txt
- 해당 섹션에 명시된 작업만 구현한다.
- 다음 섹션 작업을 미리 구현하지 않는다.
- 타 팀원 담당 영역을 수정하지 않는다.
- 구현 완료 후 관련 문서를 최신화한다.
- 사용자에게 응답하기 전 docs/rw/공통 진행사항.md를 최신화한다.
```

### 5.3 섹션 3, 5, 7, 9, 11: 검토 및 패치 단계

홀수 검토 섹션은 직전 구현 섹션만 검토한다.

예시:

```txt
섹션 3 → 섹션 2 검토
섹션 5 → 섹션 4 검토
섹션 7 → 섹션 6 검토
섹션 9 → 섹션 8 검토
섹션 11 → 섹션 10 검토
```

검토 섹션에서 해야 할 일:

```txt
1. 직전 구현 섹션의 파일 검토
2. 요구사항 충족 여부 확인
3. 소유권 위반 여부 확인
4. API/DB/공통 계약 충돌 여부 확인
5. 오류가 있으면 해당 부분만 패치
6. 오류가 없으면 다음 섹션 진행 가능 보고
7. 팀원이름_디버그 리포트.md 작성 또는 최신화
8. docs/rw/공통 진행사항.md 최신화
```

검토 섹션에서는 사용자의 명시적 지시 없이 다음 구현 섹션으로 임의 점프하지 않는다.

### 5.4 섹션 12: 전체 심층 검증 및 최종 보고

섹션 12에서는 섹션 1~11 전체를 심층 검증한다.

반드시 수행할 일:

4. docs/rw/충돌 이슈.md의 관련 이슈 확인
5. docs/rw/공통 진행사항.md와 실제 코드 불일치 여부 확인
6. 최종 테스트 또는 구조 검증 수행

## 6. 충돌 이슈 기록 규칙

### 6.1 충돌 이슈가 필요한 경우

다음 경우에는 반드시 `docs/rw/충돌 이슈.md`에 기록한다.

```txt
- 다른 팀원의 폴더를 수정해야 하는 경우
- shared_contracts의 스키마를 바꿔야 하는 경우
- API 응답 형식 변경이 필요한 경우
- Firebase DB 구조 변경이 다른 팀원에게 영향을 주는 경우
- 공공데이터 응답 구조와 백엔드 인터페이스가 맞지 않는 경우
- Flutter UI에서 필요한 필드가 백엔드/API 계약에 없는 경우
- 비콘/RSSI 데이터 구조가 UI 또는 백엔드 계약과 충돌하는 경우
- 요구사항명세서와 4월 구현범위 수정안 사이 해석 충돌이 생기는 경우
```

### 6.2 충돌 이슈 기록 형식

`docs/rw/충돌 이슈.md`에는 아래 형식을 사용한다.

```md
## CONFLICT-YYYYMMDD-HHMM-담당자명-번호

- 기록자: 심현석의 에이전트
- 작성날짜: 2026년 04월 18일 14시 32분
- 관련 팀원: 김도성
- 관련 파일/디렉터리:
  - backend/api/app/schemas/bus_info.py
  - services/public_data/public_data_client/schemas.py
- 충돌 유형: API 계약 충돌 / 담당 범위 충돌 / DB 스키마 충돌 / 파일 소유권 충돌
- 충돌 내용:
  - 구체적으로 어떤 구현을 하려 했는지 작성한다.
  - 왜 다른 팀원의 담당 범위와 충돌하는지 작성한다.
  - 어떤 요구사항과 연결되는지 작성한다.
- 현재 조치:
  - 개발 중단
  - 사용자에게 협의 필요성 안내
- 필요한 협의:
  - 어떤 팀원과 무엇을 합의해야 하는지 작성한다.
- 사용자 강제 진행 여부: 아니오
- 해결 상태: 미해결
```

### 6.3 충돌 발생 시 사용자에게 반드시 할 말

충돌 이슈를 기록한 뒤 사용자에게 다음과 같은 형식으로 알려야 한다.

```txt
**이 작업은 김도성 담당 영역과 충돌 가능성이 있으므로 더 이상 개발을 진행할 수 없습니다. 반드시 김도성 팀원에게 연락하여 공공데이터 표준 응답 형식과 백엔드 수신 인터페이스를 논의한 뒤 진행해야 합니다.**
```

### 6.4 사용자가 강제로 진행을 요구하는 경우

충돌이 해결되지 않았는데 사용자가 개발을 강요하면, 에이전트는 다음을 수행한다.

```txt
1. docs/rw/충돌 이슈.md에 "충돌 미해결 상태에서 사용자 요청으로 진행"이라고 명시한다.
2. 수정한 파일과 변경 내용을 상세히 기록한다.
3. 어떤 팀원의 영역과 충돌 가능성이 있는지 다시 기록한다.
4. 병합 시 필요한 주의사항을 기록한다.
5. 그 후 최소 범위로만 작업한다.
```

기록 예시:

```md
- 사용자 강제 진행 여부: 예
- 강제 진행 사유: 사용자가 충돌 가능성을 인지한 상태에서 개발 진행을 명시적으로 요청함
- 강제 진행 내용:
  - backend/api/app/schemas/bus_info.py에 임시 필드 추가
- 병합 시 주의:
  - 김도성의 services/public_data 응답 스키마와 반드시 재조정 필요
```

### 6.5 충돌 해결 시 삭제 금지

충돌 이슈가 해결되어도 기존 기록을 삭제하지 않는다.

해결 시 다음을 수행한다.

```txt
1. 기존 이슈 제목 또는 본문에 취소선을 적용한다.
2. 해결 일시를 기록한다.
3. 해결 기록자를 남긴다.
4. 어떤 팀원과 어떻게 합의했는지 작성한다.
5. 최종적으로 어떤 파일/계약이 기준이 되었는지 작성한다.
```

예시:

```md
~~## CONFLICT-20260418-1432-심현석-001~~

### 해결 기록

- 해결 기록자: 심현석의 에이전트
- 해결날짜: 2026년 04월 19일 10시 15분
- 협의 팀원: 김도성
- 해결 내용:
  - busArrivals 표준 응답의 공식 계약은 `packages/shared_contracts/api/bus_arrivals.response.schema.json`을 기준으로 삼고, `services/public_data/examples/mock_bus_arrivals.json`은 해당 계약을 따르는 검증용 샘플로 사용하기로 함.
  - backend/api는 해당 응답을 그대로 수신하는 gateway 역할만 수행하기로 함.
- 최종 기준 파일:
  - services/public_data/examples/mock_bus_arrivals.json
  - packages/shared_contracts/api/bus_arrivals.response.schema.json
- 해결 상태: 해결됨
```

## 14. 4월 말 MVP 기준

4월 말까지 프로젝트 전체가 달성해야 할 수준은 완성형 서비스가 아니라 다음 수준의 통합 가능한 골격이다.

```txt
- 사용자 앱 기본 UI가 존재한다.
- STT/TTS 기본 호출 구조가 존재한다.
- 접근성 UI 원칙이 반영된다.
- BLE/RSSI/방향 센서 모듈 골격이 존재한다.
- FastAPI 백엔드와 Firebase 연동 골격이 존재한다.
- 지오펜싱 판별 API 골격이 존재한다.
- FCM 알림 전송 구조가 존재한다.
- rideRequests 기반 기사-승객 매칭 데이터 흐름이 존재한다.
- 공공데이터 API 연동 모듈과 저상버스 필터링 구조가 존재한다.
- AI 비전 데이터 수집 및 모델 리서치 문서가 존재한다.
- 헤드트래킹과 공간음향은 향후 확장 프레임으로만 존재한다.
```

## 15. 마지막 확인 체크리스트

작업 시작 전 에이전트는 아래 항목을 스스로 확인한다.

```txt
[ ] 나는 어느 팀원의 에이전트인지 명확히 알고 있다.
[ ] 이 공통 프롬프트를 읽었다.

[ ] 4월 개인별 구현범위 수정안을 확인했다.
[ ] 내 팀원 전용 에이전트 필독사항을 확인했다.
[ ] 내가 수정할 파일이 내 담당 영역인지 확인했다.
[ ] 충돌 가능성이 있는지 확인했다.

```

위 항목 중 하나라도 불명확하면 개발을 시작하지 않는다.

## 선행 작업 의존성 처리 원칙

모든 AI 에이전트는 섹션 1을 시작하기 전에 프로젝트의 `docs/rw/선행작업의존성 정리.md`를 반드시 읽어야 한다.

### 1. 선행 작업물이 필요한 경우

자신의 섹션 작업을 수행하기 위해 다른 팀원의 선행 작업물, API 응답, mock 데이터, DB 스키마, UI 계약, 센서 인터페이스, 공공데이터 표준 JSON 등이 먼저 필요하다고 판단되면 임의로 대체 구현하거나 추정 구현하지 않는다.

이 경우 해당 섹션 작업을 즉시 중단하고 `docs/rw/충돌 이슈.md`에 다음 내용을 기록한다.

```txt
- 기록자:
- 작성날짜:
- 현재 작업 섹션:
- 필요한 선행 작업물:
- 선행 작업 담당 팀원:
- 현재 작업을 계속할 수 없는 이유:
- 임의 구현 시 발생 가능한 충돌:
- 요청해야 할 선행 작업 내용:
- 해당 선행 작업 완료 전까지 금지해야 할 작업:
```

기록 후 사용자에게 반드시 다음 형식으로 보고한다.

```txt
**현재 작업은 [팀원명]의 선행 작업물인 [구체적 작업물]이 먼저 필요하므로 더 이상 진행할 수 없습니다. 반드시 [팀원명]에게 연락하여 해당 선행 작업물을 먼저 생성하도록 요청한 뒤 진행해야 합니다.**
```

### 2. docs/rw/선행작업의존성 정리.md 확인 의무

각 에이전트는 자신의 섹션이 `docs/rw/선행작업의존성 정리.md`에 후행 섹션으로 명시되어 있는지 확인해야 한다.

```txt
- 후행 섹션으로 명시되어 있고 선행 섹션이 (미구현)이면 선행 산출물이 필요한 하위 작업만 보류한다. UI shell/placeholder/mock/자기 독립 로직은 허용하지만 shared contract/API/DB/status enum 임의 확정은 금지한다.
- 선행 섹션이 (구현완료_날짜)라면 실제 구현물 2차 확인
- 실제 구현물이 부실하거나 문서와 다르면 타 팀원 파일 수정 금지
- 문제 발견 시 docs/rw/충돌 이슈.md 기록 후 사용자에게 보고
```

### 3. 선행 섹션 구현 완료 시 최신화 의무

자신이 구현한 섹션이 `docs/rw/선행작업의존성 정리.md`에 선행 섹션으로 명시되어 있다면, 섹션 종료 후 반드시 해당 섹션 상태를 다음 형식으로 최신화한다.

```txt
(미구현) → (구현완료_YYYY-MM-DD HH:MM)
```

이 최신화를 하지 않은 경우, 후행 에이전트는 해당 선행 작업을 완료된 것으로 간주하지 않는다.
