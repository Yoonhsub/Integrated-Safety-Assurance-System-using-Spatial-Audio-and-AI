# docs/read/PULL_REQUEST_RULES.md

> 이 문서는 MOBI 프로젝트의 Pull Request 작성, 리뷰, 병합 기준을 정의한다.  
> 팀원과 AI 에이전트는 PR을 만들기 전에 이 문서를 확인해야 한다.

---

## 1. PR 기본 원칙

```txt
작은 PR을 만든다.
담당자와 섹션을 명확히 표시한다.
자기 담당 영역만 수정한다.
충돌 가능성은 숨기지 않는다.
문서 최신화 여부를 체크한다.
```

PR 하나에는 가능한 한 하나의 섹션 또는 하나의 문서 변경만 포함한다.

---

## 2. PR 제목 형식

```txt
[담당자][Section XX] 작업 요약
```

예시:

```txt
[심현석][Section 02] FastAPI/Firebase 기본 구조 구현
[윤현섭][Section 04] 사용자 앱 메인 UI 구현
[안준환][Section 06] RSSI 거리 추정 로직 구현
[김도성][Section 02] 공공데이터 API 조사 및 mock 기준 작성
[공통][Docs] 선행작업 의존성 규칙 보강
```

문서 작업처럼 섹션 번호가 없는 경우:

```txt
[공통][Docs] docs/read/CONTRIBUTING.md 작성
[공통][Chore] .gitignore 추가
```

```md
## 담당 정보

- 담당 팀원:

- 관련 섹션:
- 브랜치명:

-

- [ ] 자기 담당 범위만 수정했다.
- [ ] 타 팀원 담당 영역을 수정하지 않았다.
- [ ] 타 팀원 영역 수정이 필요한 경우 docs/rw/충돌 이슈.md에 기록했다.

## 선행작업 의존성 확인

- [ ] docs/rw/선행작업의존성 정리.md를 확인했다.

## 충돌 이슈

- [ ] 관련 없음
```

## 4. PR 생성 전 체크리스트

```txt
[ ] main 최신 상태를 반영했다.
[ ] 브랜치명이 docs/read/BRANCH_STRATEGY.md 규칙을 따른다.
[ ] 커밋 메시지가 docs/read/COMMIT_CONVENTION.md 규칙을 따른다.
[ ] 자기 담당 영역만 수정했다.
[ ] docs/rw/선행작업의존성 정리.md를 확인했다.
[ ] 충돌 가능성이 있으면 docs/rw/충돌 이슈.md에 기록했다.
[ ] docs/rw/공통 진행사항.md를 최신화했다.
[ ] 검토/패치 섹션이면 디버그 리포트를 최신화했다.
[ ] 섹션 12라면 최종 개발 보고서를 작성했다.
[ ] 불필요한 임시 파일을 포함하지 않았다.
[ ] API/DB/mock 구조를 임의로 확정하지 않았다.
```

## 5. 리뷰 기준

리뷰어는 아래 항목을 확인한다.

```txt
[ ] PR 제목이 명확하다.
[ ] 담당자와 섹션이 표시되어 있다.
[ ] 수정 파일이 담당 범위 안에 있다.
[ ] 타 팀원 영역 수정이 있다면 충돌 이슈가 기록되어 있다.
[ ] 선행작업 의존성이 필요한 경우 처리 규칙을 따랐다.
[ ] 문서 최신화가 누락되지 않았다.
[ ] 코드/문서가 요구사항명세서와 충돌하지 않는다.
[ ] 4월 개인별 구현범위 수정안과 충돌하지 않는다.
[ ] 헤드트래킹 등 보류 기능을 4월 구현으로 잘못 넣지 않았다.
[ ] 김도성/심현석 공공데이터 역할 분리가 지켜졌다.
```

## 6. 병합 가능 조건

아래 조건을 모두 만족하면 병합 가능하다.

```txt
[ ] PR이 자기 담당 범위 안의 변경만 포함한다.
[ ] 문서 필수 최신화가 완료되었다.
[ ] 충돌 이슈가 없거나, 충돌 이슈가 해결 또는 명확히 기록되어 있다.
[ ] 선행작업 의존성이 충족되었다.
[ ] reviewer가 최소 1명 확인했다.
[ ] validate_architecture.py가 통과한다.
```

가능하면 다음 검증을 실행한다.

```bash
python scripts/validate_architecture.py
```

## 7. 병합 보류 조건

아래 중 하나라도 해당하면 병합을 보류한다.

```txt
- main 브랜치 직접 수정
- 담당자 또는 섹션이 불명확함
- 타 팀원 영역 수정이 있는데 충돌 이슈 기록 없음
- 선행 작업물이 미구현인데 후행 구조를 임의 확정함
- docs/02_4월_개인별_구현범위_수정안.md와 충돌함
- docs/rw/공통 진행사항.md 최신화 누락
- 검토/패치 섹션인데 디버그 리포트 누락
- 섹션 12인데 최종 개발 보고서 누락
- docs/rw/API_CONTRACTS.md 또는 docs/rw/DATA_SCHEMA.md와 충돌함
```

## 8. 충돌 이슈가 있는 PR

충돌 이슈가 있는 PR은 본문에 반드시 다음을 명시한다.

```txt
Related conflict: CONFLICT-YYYYMMDD-HHMM-담당자명-번호
Conflict status:
- 미해결
- 논의 중
- 사용자 강행 요청으로 진행
- 해결됨
```

미해결 충돌이 있는 PR은 원칙적으로 병합하지 않는다.  
단, 사용자의 명시적 요청으로 강행한 경우에는 `docs/rw/충돌 이슈.md`에 그 사실이 기록되어 있어야 하며, PR에도 병합 위험을 적어야 한다.

## 9. 선행작업 의존성 관련 PR

선행 섹션을 완료한 PR은 반드시 `docs/rw/선행작업의존성 정리.md`를 최신화해야 한다.

예시:

```txt
김도성 섹션 2, 3 (미구현)
→ 김도성 섹션 2, 3 (구현완료_2026-04-18 14:32)
```

후행 섹션 PR은 반드시 선행 산출물 확인 결과를 적는다.

```txt
선행 산출물 확인:
- 김도성 섹션 2, 3 구현완료 상태 확인
- services/public_data/examples/mock_bus_arrivals.json 확인
- lowFloor, congestion 필드 확인
```

## 10. 팀원별 PR 주의사항

### 10.1 심현석

```txt
- 공공데이터 API 직접 구현 금지
- 김도성의 표준 JSON을 임의 변경 금지
- Firebase/DB 스키마 변경 시 docs/rw/DATA_SCHEMA.md와 shared_contracts 확인
- rideRequests 상태값 변경 시 윤현섭 기사 앱 UI 영향 확인
```

### 10.2 윤현섭

```txt
- FastAPI 내부 로직 직접 수정 금지
- Firebase 내부 구조 직접 변경 금지
- 공공데이터 mock JSON 임의 확정 금지
- 백엔드/API 응답 구조가 필요하면 docs/rw/선행작업의존성 정리.md 확인
```

### 10.3 안준환

```txt
- 4월에는 헤드트래킹 구현 금지
- Flutter UI 직접 수정 금지
- FastAPI/Firebase 직접 수정 금지
- BLE/RSSI/방향 센서 모듈 안에서만 구현
```

### 10.4 김도성

```txt
- FastAPI 백엔드 라우트 직접 수정 금지
- Flutter UI 직접 수정 금지
- 비콘 거리 테스트 계획과 공간음향 방향 테스트 계획 담당 아님
- 공공데이터 표준 JSON을 변경하면 심현석/윤현섭 영향 명시
```

## 11. PR 병합 후 해야 할 일

PR이 병합되면 다음을 확인한다.

```txt
[ ] 브랜치 삭제 여부 결정
[ ] docs/rw/공통 진행사항.md에 병합 사실 기록
[ ] 관련 충돌 이슈 상태 최신화
[ ] docs/rw/선행작업의존성 정리.md 상태 최신화
[ ] 후행 담당 팀원에게 필요한 경우 알림
```

---

## 12. 금지 사항

```txt
- "일단 합치고 나중에 고치자" 식의 대형 PR
- 여러 팀원 영역이 섞인 PR
- 충돌 이슈를 숨긴 PR
- 선행 작업물이 없는데 후행 API/DB 구조를 확정한 PR
- 디버그 리포트 없이 검토 섹션 완료 처리
- 최종 보고서 없이 섹션 12 완료 처리
```

## 13. 최종 원칙

```txt
PR은 작게.
리뷰는 명확하게.
충돌은 기록하게.
선행 작업물은 확인하게.
병합은 신중하게.
```
