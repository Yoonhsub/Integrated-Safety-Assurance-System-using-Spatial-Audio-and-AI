# 윤현섭 디버그 리포트

## 1. 문서 목적

이 문서는 윤현섭 담당 범위인 `apps/passenger_app/**`, `apps/driver_app/**` 개발 중 발생한 문제, 검토 결과, 해결 여부, 테스트 미수행 사항을 정리하기 위한 디버그 리포트이다.

윤현섭 담당 범위는 다음으로 한정하였다.

- `apps/passenger_app/**`
- `apps/driver_app/**`

다음 영역은 임의 수정하지 않았다.

- `backend/api/**`
- `infrastructure/firebase/**`
- `services/public_data/**`
- `packages/mobile_sensors/**`
- `ai_vision/**`
- `packages/shared_contracts/**`

---

## 2. 전체 디버그 요약

전체 개발은 Section 01부터 Section 12까지 순차적으로 진행하였다.

주요 검토 결과는 다음과 같다.

- `passenger_app` 실행 확인
- `driver_app` 실행 확인
- `flutter analyze` 기준 정적 분석 문제 없음 확인
- 윤현섭 담당 범위 외 파일 수정 없음 확인
- API/DB/schema/status enum/shared contracts 임의 확정 없음 확인
- STT/TTS는 초기 stub 구조에서 실제 플러그인 wrapper 구조로 보강
- 승객 앱 백엔드 상태 카드는 초기 placeholder 구조에서 mock 렌더링 구조로 보강
- 기사 앱 rideRequests 관련 기능은 실제 API 연동 없이 UI shell 및 placeholder 수준으로 유지

## 4. Flutter 앱 구현 중 남은 stub, mock, placeholder 항목

### 4.1 passenger_app

승객 앱에 남아 있는 stub, mock, placeholder 항목은 다음과 같다.

#### STT/TTS

- 초기 구현은 `VoiceGuideService` stub 구조였다.
- 이후 `speech_to_text`, `flutter_tts` 기반 실제 plugin wrapper로 보강하였다.
- 다만 실기기 마이크 권한, 브라우저별 음성 인식 동작, 음성 인식 실패 상황은 추가 검증이 필요하다.

#### 안전 상태

- `/geofence/check` 실제 API 계약이 완전히 확정되지 않았으므로 mock 상태 렌더링으로 처리하였다.
- 실제 위치 기반 안전 상태 판단 로직은 후속 연동이 필요하다.

#### 버스 도착 정보

- `/bus-info/stops/{stopId}/arrivals` 실제 공공데이터 mock 기준이 확정되기 전까지 mock 도착 정보로 표시하였다.
- 실제 버스 번호, 도착 시간, 남은 정류장, 저상버스 여부 등은 임의 확정하지 않았다.

#### 탑승 요청 상태

- `/ride-requests` 파이프라인이 확정되기 전까지 mock 요청 상태 또는 “요청 전” 상태로 표시하였다.
- 실제 요청 생성, 요청 상태 변경, 요청 완료 처리는 후속 작업이 필요하다.

### 4.2 driver_app

기사 앱에 남아 있는 stub, mock, placeholder 항목은 다음과 같다.

#### 탑승 요청 목록

- 실제 rideRequests API를 호출하지 않는다.
- 정적 예시 탑승 요청 카드만 표시한다.
- 실제 요청 목록 조회는 후속 작업이다.

#### 요청 처리

- “요청 확인 준비 중” 버튼은 UI shell이다.
- 실제 요청 수락, 확인, 거절, 상태 변경 로직은 구현하지 않았다.

#### rideRequests 필드

다음 필드는 임의 확정하지 않았다.

- `requestId`
- `userId`
- `stopId`
- `routeId`
- `busNo`
- `status enum`

## 5. 실제 테스트하지 못한 항목

다음 항목은 현재 개발 환경 또는 선행 모듈 미확정으로 인해 실제 테스트하지 못했다.

### 5.1 실기기 접근성 테스트

- Android TalkBack 실제 읽기 순서
- iOS VoiceOver 실제 읽기 순서
- 큰 글씨 설정
- 고대비 모드
- 실제 터치 영역 사용성

### 5.2 실제 STT/TTS 환경 테스트

- Android 실기기 마이크 권한 허용 후 음성 인식
- iOS 실기기 마이크 권한 허용 후 음성 인식
- 브라우저별 Web Speech 지원 차이
- 음성 인식 실패 처리
- TTS 출력 음성 품질
- 음성 출력 중 중복 입력 처리

### 5.3 실제 백엔드 API 테스트

다음 API는 실제 서버 연동 테스트를 완료하지 못했다.

- `/geofence/check`
- `/bus-info/stops/{stopId}/arrivals`
- `/ride-requests`

사유는 다음과 같다.

- API 계약 및 응답 구조가 완전히 확정되지 않은 상태였음
- 윤현섭 담당 범위 밖의 backend/schema/shared contracts를 임의 수정할 수 없었음
- 따라서 passenger_app 내부 mock 렌더링으로 대체함

### 5.4 실제 rideRequests 기사 앱 테스트

- 실제 탑승 요청 목록 수신
- 실제 요청 확인 처리
- 요청 상태 변경
- 실시간 업데이트
- 기사 앱과 승객 앱 간 요청 동기화

위 항목은 rideRequests 파이프라인 확정 후 후속 테스트가 필요하다.

## 6. 주요 해결 내역

### 6.1 한글 깨짐 UI 정리

초기 `home_page.dart` 일부 문구가 깨져 보이는 문제가 있었다.

해결 내용:

- 승객 앱 홈 화면 문구 재작성
- 기사 앱 홈 화면 문구 재작성
- MVP 안내 문구 추가
- 사용자와 기사 앱 간 문구 톤 통일

### 6.2 윤현섭 범위 외 파일 수정 방지

개발 중 계속 다음 원칙을 유지하였다.

- `apps/passenger_app/**` 안에서 승객 앱 작업
- `apps/driver_app/**` 안에서 기사 앱 작업
- 그 외 공통 계약, 백엔드, 인프라, 공공데이터, BLE, AI Vision 영역은 수정하지 않음

### 6.3 API 필드 및 status enum 임의 확정 방지

다음 구조는 임의 확정하지 않았다.

- geofence 응답 구조
- 공공데이터 mock 필드
- rideRequests 요청 필드
- rideRequests status enum
- DB schema
- shared contracts

### 6.4 PR 리뷰 피드백 반영

리뷰 피드백에 따라 다음 항목을 보강하였다.

- `VoiceGuideService`를 실제 STT/TTS plugin wrapper 구조로 개선
- 승객 앱 홈 화면에서 `BackendApiClient` 기반 mock 상태 렌더링 적용
- 최종 개발 보고서 작성
- 디버그 리포트 작성

다만 STT/TTS 실제 연동 보강과 백엔드/mock 상태 렌더링 보강은 디버그 리포트 작성 시점 기준으로 코드 수정 및 커밋까지 완료되었으나, 아직 메인 브랜치 병합 전 상태이다.

## 7. 최종 디버그 판정

윤현섭 담당 범위 내에서 확인한 결과는 다음과 같다.

- `apps/passenger_app` 정적 분석 문제 없음
- `apps/driver_app` 정적 분석 문제 없음
- 승객 앱 Chrome 실행 확인
- 기사 앱 Chrome 실행 확인
- 윤현섭 범위 외 파일 수정 없음
- API/DB/schema/status enum/shared contracts 임의 확정 없음
- 남은 항목은 선행 모듈 확정 후 실제 연동 및 실기기 테스트 필요

따라서 현재 기준 윤현섭 담당 앱 UI 범위의 디버그 기록은 완료로 판단한다.
