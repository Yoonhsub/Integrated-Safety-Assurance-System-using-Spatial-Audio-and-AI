# 공통 진행사항

## 공통 진행사항

> 이 문서는 MOBI 프로젝트의 팀원별 AI 에이전트가 **섹션 단위 작업을 완료할 때마다 반드시 최신화해야 하는 공통 진행 로그**이다.  
> 각 에이전트는 반드시 자기 팀원 기록 공간에만 작성해야 하며, 다른 팀원의 기록 공간을 수정하거나 삭제해서는 안 된다.

---

## 1. 문서의 목적

본 프로젝트는 4명의 팀원이 각자 생성형 AI를 활용하여 병렬 개발하는 구조이다.  
따라서 다음 사항을 투명하게 기록해야 한다.

```txt
- 어느 팀원의 에이전트가 작업했는가
- 어떤 섹션을 완료했는가
- 어떤 파일을 수정했는가
- 이전 섹션에서 언급된 이슈를 어떻게 처리했는가
- 남은 이슈는 무엇인가
- 충돌 이슈가 있었는가
- 현재 담당 파트의 진척도는 몇 %인가
- 다음 섹션으로 넘어가도 되는가
```

이 문서는 팀원 간 병합 충돌, 책임 불명확성, 작업 누락을 줄이기 위한 공통 기록 체계이다.

---

## 2. 절대 작성 원칙

### 2.1 자기 팀원 기록 공간만 수정한다

각 에이전트는 자신이 담당하는 팀원의 기록 공간에만 작성해야 한다.

```txt
심현석의 에이전트 → "심현석의 에이전트 기록 공간"만 수정
윤현섭의 에이전트 → "윤현섭의 에이전트 기록 공간"만 수정
안준환의 에이전트 → "안준환의 에이전트 기록 공간"만 수정
김도성의 에이전트 → "김도성의 에이전트 기록 공간"만 수정
```

다른 팀원의 기록 공간을 임의로 고치거나 정리하거나 삭제해서는 안 된다.

---

### 2.2 섹션 하나가 끝날 때마다 반드시 기록한다

다음 작업이 끝날 때마다 반드시 이 문서를 최신화한다.

```txt
- 섹션 1 사전 검토 완료
- 섹션 2 직접 구현 완료
- 섹션 3 검토/패치 완료
- 섹션 4 직접 구현 완료
- 섹션 5 검토/패치 완료
- 섹션 6 직접 구현 완료
- 섹션 7 검토/패치 완료
- 섹션 8 직접 구현 완료
- 섹션 9 검토/패치 완료
- 섹션 10 직접 구현 완료
- 섹션 11 검토/패치 완료
- 섹션 12 최종 심층검증 완료
```

섹션 작업을 완료했는데 이 문서를 최신화하지 않았다면 해당 섹션은 완료된 것으로 보지 않는다.

---

### 2.3 구체적 날짜와 시간을 기록한다

기록 시점의 구체적인 년, 월, 일, 시, 분을 남긴다.

예시:

```txt
기록일시: 2026년 4월 18일 14시 32분
```

시간대는 별도 언급이 없다면 대한민국 표준시, KST 기준으로 기록한다.

---

### 2.4 진행률은 담당 파트 기준으로 작성한다

진척도는 전체 프로젝트 기준이 아니라 **자기 담당 파트의 4월 구현 범위 기준**으로 작성한다.

예시:

```txt
현재 담당 파트 진척도: 35%
```

진척도 산정 시 고려할 요소:

```txt
- 해당 팀원의 12섹션 중 몇 섹션을 완료했는가
- 구현 섹션이 실제로 코드/문서/테스트 형태로 남았는가
- 검토 섹션에서 발견된 오류가 해결되었는가
- 충돌 이슈로 막힌 작업이 있는가
```

---

### 2.5 충돌이 있으면 docs/rw/충돌 이슈.md와 연결한다

충돌 가능성이 있었다면 반드시 `docs/rw/충돌 이슈.md`의 이슈 번호를 함께 적는다.

예시:

```txt
관련 충돌 이슈: CONFLICT-20260418-1530-심현석-002
```

충돌이 없었다면 다음과 같이 적는다.

```txt
관련 충돌 이슈: 없음
```

---

### 2.6 섹션 종료 시 사용자에게 GitHub 업로드를 권유한다

각 섹션 완료 후 사용자에게 반드시 아래 문구를 굵게 전달해야 한다.

```txt
**지금 섹션 X 끝났습니다. 캡스톤 과목 점수 획득 및 충돌 병목 최소화를 위해 지금까지 진행상황 깃헙에 올리시는 것을 권유드립니다.**
```

여기서 `X`는 실제 완료한 섹션 번호로 바꾼다.

## 3. 공통 기록 양식

각 팀원 기록 공간에 아래 양식을 복사해서 사용한다.

```md
### 기록 0000

- 기록자:
- 기록일시:
- 완료 섹션:
- 섹션 유형:
  - [ ] 사전 검토
  - [ ] 직접 구현
  - [ ] 검토/패치
  - [ ] 최종 심층검증
- 이번 섹션의 목표:
- 이번 섹션에서 수행한 작업:
- 수정/생성한 주요 파일:
- 이전 섹션에서 언급된 이슈:
- 이전 이슈 처리 결과:
- 이번 섹션에서 새로 발견한 이슈:
- 관련 충돌 이슈:
- 작성/최신화한 문서:
- 테스트 또는 검증 결과:
- 현재 담당 파트 진척도:
- 다음 섹션 진행 가능 여부:
- 다음 섹션에서 해야 할 일:
- 사용자에게 전달한 GitHub 업로드 권유 문구:
```

---

## 4. 기록 번호 관리 규칙

각 팀원 기록 공간 안에서 독립적으로 번호를 관리한다.

예시:

```txt
심현석 기록 공간: 기록 0001, 기록 0002, 기록 0003 ...
윤현섭 기록 공간: 기록 0001, 기록 0002, 기록 0003 ...
안준환 기록 공간: 기록 0001, 기록 0002, 기록 0003 ...
김도성 기록 공간: 기록 0001, 기록 0002, 기록 0003 ...
```

다른 팀원의 기록 번호와 일치해도 상관없다.  
단, 같은 팀원 기록 공간 안에서는 번호를 중복해서 사용하지 않는다.

- 실제 구현물 2차 확인 결과: 해당 없음(섹션 1).
- docs/rw/선행작업의존성 정리.md 최신화 여부: 미실시(섹션 1은 미구현 상태 변경 없음).
- 후행 작업 가능 여부: 김도성 후속 섹션 진행은 외부 선행 의존이 없어 즉시 가능. 후행 팀원(윤현섭/심현석)은 자기 섹션 1~5/1~9를 우선 진행하면 되고, 김도성 섹션 2,3 또는 6,7 완료 시점에 본 문서 최신화 예정.

## 9. 초기 상태

```txt
문서 생성 상태: 초기 템플릿 생성 완료
현재 등록된 진행 기록: 없음
사용 시작 시점: 각 팀원의 AI 에이전트가 섹션 1을 시작하는 시점
```

이 문서는 프로젝트 초기 문서 생성 단계에서는 단순 템플릿으로 유지하고, 실제 팀원별 개발 섹션이 시작된 뒤부터 본격적으로 최신화한다.

## docs/rw/선행작업의존성 정리.md 기록 연동 규칙

각 에이전트는 섹션 종료 시 자신의 작업이 `docs/rw/선행작업의존성 정리.md`와 관련 있는지 확인하고, 관련이 있다면 자기 팀원 기록 공간에 반드시 다음 항목을 남긴다.

```txt
- docs/rw/선행작업의존성 정리.md 확인 여부:
- 관련 의존성 ID:
- 내 섹션이 선행 섹션인지/후행 섹션인지:
- 선행 섹션 상태:
- 실제 구현물 2차 확인 결과:
- docs/rw/선행작업의존성 정리.md 최신화 여부:
- 후행 작업 가능 여부:
```

자신이 구현한 섹션이 선행 섹션으로 명시되어 있는데 이 문서를 최신화하지 않았다면, 해당 섹션은 후행 에이전트 기준에서 완료된 것으로 보지 않는다.
