# 충돌 이슈

> 이 문서는 MOBI 프로젝트의 AI 병렬 개발 과정에서 발생할 수 있는 **역할 충돌, 파일 소유권 충돌, API 계약 충돌, 데이터 스키마 충돌, 병합 위험**을 기록하기 위한 최상위 관리 문서이다.  
> 모든 AI 에이전트는 충돌 가능성을 발견하면 임의로 개발을 계속하지 말고, 이 문서에 먼저 기록해야 한다.

---

## 1. 문서의 목적

본 프로젝트는 4명의 팀원이 각자 생성형 AI를 활용하여 병렬 개발하는 구조이다.  
따라서 다음 문제가 발생할 수 있다.

```txt
- 한 에이전트가 다른 팀원의 담당 파일을 수정하는 문제
- 서로 다른 에이전트가 같은 API 응답 구조를 다르게 정의하는 문제
- Firebase Realtime Database 스키마를 각자 다르게 해석하는 문제
- Flutter 앱, FastAPI 백엔드, 공공데이터 모듈, 센서 모듈 간 인터페이스 불일치
- 병합 시점에 대규모 Git 충돌이 발생하는 문제
- 사용자 요청으로 충돌이 해결되지 않은 상태에서 강행 개발되는 문제
```

이 문서는 위와 같은 충돌을 **숨기지 않고 기록하며**, 해결 후에도 삭제하지 않고 히스토리를 보존하기 위해 존재한다.

---

## 2. 절대 원칙

### 2.1 충돌 이슈는 절대 삭제하지 않는다

충돌이 해결되었더라도 해당 기록은 삭제하지 않는다.  
해결된 이슈는 반드시 **취소선**을 적용하고, 그 아래에 해결 내역을 추가한다.

예시:

```md
~~### CONFLICT-20260418-1432-심현석-001: /rideRequests status 값 정의 충돌~~

#### 해결 내역
- 해결일시:
- 해결자:
- 해결 방식:
- 최종 기준:
```

---

### 2.2 충돌 가능성이 있으면 개발을 멈춘다

에이전트가 자신의 담당 범위를 구현하는 과정에서 다른 팀원의 담당 영역을 수정해야 하거나, 다른 팀원의 설계와 충돌 가능성이 있다고 판단하면 즉시 개발을 멈춘다.

개발을 멈춘 뒤 해야 할 일은 다음과 같다.

```txt
1. 이 문서의 "미해결 충돌 이슈"에 충돌 내용을 기록한다.
2. 사용자에게 충돌 가능성을 명확히 보고한다.
3. 관련 팀원의 이름을 구체적으로 언급한다.
4. 사용자가 해당 팀원과 논의하도록 안내한다.
5. 충돌이 해결되기 전까지 임의로 구현을 계속하지 않는다.
```

---

### 2.3 기록자 정보는 반드시 팀원 이름 기준으로 남긴다

AI 에이전트는 자신이 담당하는 팀원의 이름을 빌려 기록해야 한다.

예시:

```txt
기록자: 심현석의 에이전트
작성날짜: 2026년 4월 18일 14시 32분
```

허용되는 기록자 표기는 다음 4개뿐이다.

```txt
- 기록자: 심현석의 에이전트
- 기록자: 윤현섭의 에이전트
- 기록자: 안준환의 에이전트
- 기록자: 김도성의 에이전트
```

자신이 어느 팀원의 에이전트인지 명시되지 않은 상태에서는 이 문서에 기록하거나 개발을 진행해서는 안 된다.

---

### 2.4 사용자가 강행 개발을 요구한 경우도 반드시 기록한다

충돌이 해결되지 않았음에도 사용자가 개발 진행을 강하게 요구하면, 에이전트는 다음 내용을 이 문서에 반드시 남겨야 한다.

```txt
- 충돌이 해결되지 않았음
- 사용자의 명시적 요청으로 개발을 진행했음
- 어떤 파일을 수정했는지
- 어떤 팀원의 담당 범위와 충돌 가능성이 있는지
- 병합 시 어떤 문제가 생길 수 있는지
```

이 기록 없이 강행 개발을 진행해서는 안 된다.

---

### 2.5 해결된 충돌은 "해결된 충돌 이슈"로 이동하지 않고, 원래 기록에 해결 내역을 덧붙인다

충돌 이슈는 발생 당시 맥락이 중요하므로 원래 기록을 보존한다.  
해결되면 해당 제목과 핵심 충돌 설명에 취소선을 긋고, 바로 아래에 해결 내역을 작성한다.

---

## 3. 충돌로 간주해야 하는 상황

아래 상황은 모두 충돌 이슈로 기록해야 한다.

### 3.1 파일 소유권 충돌

```txt
- 심현석 에이전트가 services/public_data 내부 구현을 수정해야 하는 경우
- 김도성 에이전트가 backend/api 내부 FastAPI 라우트를 수정해야 하는 경우
- 윤현섭 에이전트가 backend/api 또는 services/public_data 내부 로직을 수정해야 하는 경우
- 안준환 에이전트가 Flutter UI 화면 또는 FastAPI 백엔드 로직을 수정해야 하는 경우
```

---

### 3.2 API 계약 충돌

```txt
- 윤현섭 앱 화면에서 필요한 응답 필드와 심현석 백엔드 응답 필드가 다를 때
- 심현석 백엔드가 기대하는 busArrivals 구조와 김도성 공공데이터 모듈의 표준 JSON이 다를 때
- 에러 응답 형식이 모듈마다 다르게 정의될 때
- status 값, enum 값, 필수 필드가 문서와 다르게 구현될 때
```

---

### 3.3 데이터 스키마 충돌

```txt
- Firebase Realtime Database 경로가 문서와 다르게 추가/변경될 때
- rideRequests 상태값이 팀원마다 다르게 정의될 때
- users, drivers, busStops, geofences, busArrivals 구조를 임의로 바꾸어야 할 때
```

---

### 3.4 4월 구현 범위 충돌

아래 항목은 특히 주의해야 한다.

```txt
- 안준환: 4월에는 헤드트래킹 구현 금지
- 심현석: 공공데이터 API 직접 구현 금지
- 김도성: 공공데이터 API 기반 버스 정보 연동 담당
- 김도성: 비콘 거리 테스트 계획 및 공간음향 방향 테스트 계획 제외
- 윤현섭: 백엔드 내부 로직, Firebase 내부 로직, 공공데이터 API 직접 구현 금지
```

위 원칙을 어겨야 하는 상황이 생기면 반드시 충돌 이슈로 기록한다.

---

## 4. 충돌 이슈 작성 양식

아래 양식을 복사해서 사용한다.

```md
### CONFLICT-YYYYMMDD-HHMM-담당자명-번호: [충돌 제목]

- 기록자:
- 작성날짜:
- 현재 작업 섹션:
- 관련 팀원:
- 관련 파일/폴더:
- 충돌 유형:
  - [ ] 파일 소유권 충돌
  - [ ] API 계약 충돌
  - [ ] 데이터 스키마 충돌
  - [ ] 4월 구현 범위 충돌
  - [ ] Git 병합 위험
  - [ ] 사용자 강행 요청
  - [ ] 기타
- 충돌 내용:
- 왜 충돌이라고 판단했는가:
- 현재 개발을 중단해야 하는 이유:
- 논의가 필요한 팀원:
- 사용자에게 전달한 경고 문구:
- 임시 우회 가능 여부:
- 임시 우회 시 위험:
- 최종 결정 전까지 금지해야 할 작업:
- 상태:
  - [ ] 미해결
  - [ ] 논의 중
  - [ ] 사용자 강행 요청으로 진행
  - [ ] 해결됨

#### 사용자 강행 요청 기록

- 사용자가 충돌 미해결 상태에서 진행을 요구했는가:
- 강행 요청 일시:
- 강행 요청 내용:
- 강행으로 수정한 파일:
- 병합 시 예상 위험:

#### 해결 내역

- 해결일시:
- 해결자:
- 협의한 팀원:
- 최종 해결 방식:
- 최종 기준 문서/파일:
- 후속 조치:
```

## 5. 사용자에게 반드시 전달해야 하는 경고 문구

충돌 이슈를 기록한 뒤에는 사용자에게 반드시 굵은 글씨로 다음 형식의 문구를 전달한다.

```txt
**현재 작업은 [팀원명] 담당 파트와 충돌 가능성이 있어 더 이상 개발을 진행할 수 없습니다. 반드시 [팀원명]에게 연락하여 이 이슈를 논의한 뒤 진행해야 합니다.**
```

예시:

```txt
**현재 작업은 김도성 담당 공공데이터 API 표준 응답 구조와 충돌 가능성이 있어 더 이상 개발을 진행할 수 없습니다. 반드시 김도성에게 연락하여 이 이슈를 논의한 뒤 진행해야 합니다.**
```

---

## 6. 미해결 충돌 이슈

아직 등록된 미해결 충돌 이슈가 없다.

새로운 충돌이 발생하면 `CONFLICT-YYYYMMDD-HHMM-담당자명-번호` 형식으로 작성한다.

---

## 7. 해결된 충돌 이슈

아직 해결된 충돌 이슈가 없다.

해결된 이슈는 원래 이슈 제목과 핵심 충돌 설명에 취소선을 적용한 뒤, 해결 내역을 아래에 추가한다.

---

## 8. 충돌 이슈 번호 관리 규칙

```txt
CONFLICT-20260418-1432-심현석-001
CONFLICT-20260418-1500-윤현섭-001
CONFLICT-20260418-1530-심현석-002
...
```

- 같은 담당자·같은 시각의 번호는 중복되면 안 된다.
- 이미 사용된 번호는 삭제된 것처럼 보이더라도 재사용하지 않는다.
- 해결된 이슈 번호도 재사용하지 않는다.
- 에이전트가 번호 판단이 어렵다면 가장 마지막 번호 다음 번호를 사용한다.

---

## 9. 초기 상태

- 문서 생성 상태: 초기 템플릿 생성 완료
- 현재 등록된 충돌 이슈: 없음
- 현재 해결된 충돌 이슈: 없음
- 이 문서는 실제 팀원별 개발 섹션이 시작된 뒤부터 본격적으로 사용한다.

## 선행 작업 의존성 미충족 처리 원칙

다른 팀원의 선행 작업물이 없으면 자기 섹션을 안전하게 구현할 수 없는 경우, 이는 충돌 이슈로 간주한다.

충돌 유형에 아래 항목을 추가하여 기록한다.

```txt
- [ ] 선행 작업 의존성 미충족
```

기록 시 반드시 포함할 내용:

```txt
- 필요한 선행 작업물
- 선행 작업 담당 팀원
- 선행 섹션 번호
- 현재 중단된 후행 섹션 번호
- 임의 구현 시 발생 가능한 API/DB/UI/파일 소유권 충돌
- 해당 선행 작업이 완료되기 전까지 금지해야 할 작업
```

사용자에게 전달해야 할 문구:

```txt
**현재 작업은 [팀원명]의 선행 작업물인 [구체적 작업물]이 먼저 필요하므로 더 이상 진행할 수 없습니다. 반드시 [팀원명]에게 연락하여 해당 선행 작업물을 먼저 생성하도록 요청한 뒤 진행해야 합니다.**
```
