# 윤현섭 V2 최종 개발 보고서

## 1. 담당 범위

윤현섭 담당 범위는 다음 앱 UI 영역으로 한정하였다.

- `apps/passenger_app/**`
- `apps/driver_app/**`

이번 V2 작업에서는 승객 앱과 기사 앱의 Flutter UI를 중심으로, 기존 4월 MVP 단계의 mock-first 화면을 실제 앱-백엔드 통합 가능한 구조로 고도화하였다.

주요 작업 방향은 다음과 같다.

- 승객 앱과 기사 앱의 `BackendApiClient` 구조 정리
- `/health` 연결 상태 표시
- 승객 앱 버스 도착 정보 API 연결 시도 구조 추가
- 승객 앱 탑승 요청 생성 및 상태 조회 구조 추가
- 기사 앱 탑승 요청 목록 조회 구조 추가
- 기사 앱 탑승 요청 수락 상태 변경 구조 추가
- 승객 앱 현재 상태 기반 음성 안내 연결
- STT/TTS 및 접근성 Semantics 검증
- geofence/safety/mobile_sensors 관련 계약 확인 및 placeholder 유지

백엔드 API, Firebase, 공공데이터 수집 모듈, BLE/RSSI 모듈, AI Vision, shared contracts 영역은 임의로 수정하지 않았다.

수정하지 않은 범위는 다음과 같다.

- `backend/**`
- `backend/api/**`
- `infrastructure/firebase/**`
- `services/public_data/**`
- `packages/mobile_sensors/**`
- `ai_vision/**`
- `packages/shared_contracts/**`

API schema, DB schema, shared contracts, status enum은 임의로 새로 정의하지 않았으며, 확인된 백엔드 계약 또는 기존 문서에 기반하여 앱 내부 표시용 모델만 구성하였다.

## 4. 미구현 기능 및 후속 작업 필요 사항

### 4.1 실제 백엔드 성공 응답 검증

V2에서는 백엔드 서버가 실행 중이지 않은 환경에서 연결 실패 fallback을 중심으로 검증하였다.

후속 작업이 필요한 항목은 다음과 같다.

- 실제 백엔드 서버 실행 상태에서 `/health` 성공 응답 검증
- 실제 `/bus-info/stops/{stopId}/arrivals` 성공 응답 검증
- 실제 `POST /ride-requests` 성공 응답 검증
- 실제 `GET /ride-requests/{requestId}` 성공 응답 검증
- 실제 `GET /drivers/{driverId}/ride-requests` 성공 응답 검증
- 실제 `PATCH /ride-requests/{requestId}/status` 성공 응답 검증

### 4.2 실제 버스 도착 정보 확장

현재 V2에서는 확인된 필드인 `busNo`, `routeId`, `congestion` 중심으로만 표시하였다.

후속 작업이 필요한 항목은 다음과 같다.

- arrivalMinutes 표시
- remainingStops 표시
- lowFloor 표시
- 정확한 도착 예정 시간 표시
- 저상버스 여부 표시
- 실제 BusArrival 전체 schema 확정 후 표시 확장

### 4.3 실제 rideRequests 처리 확장

현재 V2에서는 요청 생성, 상태 조회, 기사 목록 조회, 요청 수락 구조를 앱 내부에서 연결 가능한 형태로 구성하였다.

후속 작업이 필요한 항목은 다음과 같다.

- request body 필드 확정
- userId, stopId, routeId, busNo, targetDriverId 처리
- 실제 requestId 기반 전체 상태 흐름 검증
- ARRIVED 상태 변경
- COMPLETED 상태 변경
- CANCELLED 상태 변경
- passenger_app 상태 실시간 반영
- driver_app 요청 목록 실시간 갱신
- FCM 알림 연동

### 4.4 실제 STT/TTS 고도화

승객 앱에는 음성 입력 및 현재 상태 음성 안내 흐름을 구현하였다.

후속 작업이 필요한 항목은 다음과 같다.

- 실제 모바일 권한 기반 STT 검증
- 실제 모바일 권한 기반 TTS 검증
- 음성 인식 실패 처리 고도화
- 음성 인식 결과를 실제 목적지 검색 API와 연결
- 음성 명령 기반 자동 탑승 요청 생성
- 모바일 기기별 TTS 출력 품질 확인

### 4.5 안전 상태 및 센서 연동

현재 안전 상태 카드는 geofence/safety 이벤트와 연결 가능한 placeholder 구조로 유지하였다.

후속 작업이 필요한 항목은 다음과 같다.

- 실제 `POST /geofence/check` 호출
- 실제 위치 좌표 기반 geofence 판정
- `GeofenceCheckRequest` 요청 body 계약 확정
- `GeofenceCheckResponse` 앱 표시 확장
- `shouldSpeak`, `shouldVibrate` 앱 동작 연결
- BLE/RSSI/mobile_sensors 연동
- DirectionSensor 연동
- 실제 안전 알림 UX 검증

### 4.6 접근성 실기기 테스트

현재 Flutter `Semantics`를 적용했지만, 실제 Android TalkBack 또는 iOS VoiceOver 실기기 검증은 추가로 필요하다.

후속 작업이 필요한 항목은 다음과 같다.

- TalkBack 포커스 순서 확인
- VoiceOver 읽기 순서 확인
- 큰 글씨 환경 확인
- 고대비 모드 확인
- 터치 영역 크기 확인
- 실제 시각장애인/노약자 사용성 테스트

## 7. 충돌 이슈

현재까지 윤현섭 담당 범위 외 파일을 임의 수정하지 않았다.

확인 결과 다음 영역은 수정하지 않았다.

- `backend/**`
- `backend/api/**`
- `infrastructure/firebase/**`
- `services/public_data/**`
- `packages/mobile_sensors/**`
- `ai_vision/**`
- `packages/shared_contracts/**`

API/DB/schema/status enum도 임의로 확정하지 않았다.

특히 다음 항목은 앱에서 임의 확정하지 않고 후속 의존성으로 남겼다.

- `arrivalMinutes`
- `remainingStops`
- `lowFloor`
- ride request request body 필드
- 새로운 status enum
- geofence 요청 body
- sensor event schema
- `shouldSpeak`
- `shouldVibrate`

따라서 현재 기준 명시적인 충돌 이슈는 없다.
