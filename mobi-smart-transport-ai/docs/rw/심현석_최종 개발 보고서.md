```txt
- 전체 완료 상태: 선행작업의존 패치 완료

```

심현석 담당 백엔드 코어 영역인 FastAPI 앱 구조, Firebase lazy/mock client, geofence API, FCM notification service, rideRequests pipeline은 섹션 1~11 범위에서 구현·검토·테스트까지 완료했다.

선행작업의존 패치에서 김도성 섹션 6~7 완료와 `BusArrivalsService.get_arrivals(stop_id)` 진입점을 확인했고, `bus_info_gateway`가 RTDB cache miss 시 해당 표준 응답을 호출하도록 보완했다. backend는 여전히 공공데이터 원본 필드를 직접 해석하지 않고, `services/public_data/**` 내부 구현도 수정하지 않는다.

후속 문서 정합성 검토에서 실제 코드·테스트·shared contract 기준을 재대조한 결과, busArrivals cache 계약은 `/busArrivals/{stopId}` + `BusArrivalsResponse` 구조로 확정했다. 이에 맞춰 `DATA_SCHEMA.md`, `API_CONTRACTS.md`, `realtime_database.schema.json`의 busArrivals 설명을 실제 구현 기준으로 정합화했다.

## 4. 구현 완료 항목

### 4.1 FastAPI·Firebase 기초 구조

- `backend/api/app/main.py` 중심 FastAPI app 구조 정비
- `/health` API 확장
- CORS 설정
- 공통 서비스 예외 처리
- `FirebaseClient` lazy initialization
- Firebase 인증정보가 없어도 import/test가 깨지지 않는 mock fallback
- RTDB helper: `get`, `set`, `update`, `push`, `delete`

### 4.2 지오펜싱 API

- `POST /geofence/check`
- `SAFE`, `WARNING`, `DANGER`, `OUT_OF_AREA`, `UNKNOWN` 상태 구분
- RTDB `/geofences/{stopId}` 우선 조회
- mock geofence fallback
- polygon 내부 판별 및 거리 기반 fallback
- 사용자 최근 위치 `/users/{userId}/currentLocation` 저장
- 상태 전이 이벤트 `/systemLogs/{logId}` 기록
- 동일 alert 상태 반복 호출 시 중복 system log 방지
- 상태별 log level 보정

### 4.3 FCM 토큰·알림 구조

- 공식 FCM token path `/fcmTokens/{ownerType}/{ownerId}` 고정
- 사용자/기사 token 조회 분리
- 실제 인증정보가 없을 때 mock message id 반환
- `send_safety_alert()` helper
- `send_ride_request_notification()` helper
- notification target validation 보강

### 4.4 rideRequests 매칭 파이프라인

- `POST /ride-requests`
- `GET /ride-requests/{requestId}`
- `PATCH /ride-requests/{requestId}/status`
- `GET /drivers/{driverId}/ride-requests`
- 상태 enum: `WAITING`, `NOTIFIED`, `ACCEPTED`, `ARRIVED`, `COMPLETED`, `CANCELLED`
- 저장 경로 `/rideRequests/{requestId}`
- RTDB value 내부 `requestId` 중복 저장 방지
- 기사 알림 실패 시 rideRequest 저장 자체는 실패시키지 않도록 설계

### 4.5 bus_info_gateway public_data interface

- `GET /bus-info/stops/{stopId}/arrivals`
- RTDB cache `/busArrivals/{stopId}` 조회
- cache miss 시 김도성 `BusArrivalsService.get_arrivals(stop_id)` 표준 응답 호출
- normalized `BusArrivalsResponse` 저장 interface
- provider-specific field를 backend cache에 저장하지 않도록 테스트 보강
- 문서/스키마 기준도 `/busArrivals/{stopId}` + `BusArrivalsResponse` 구조로 정합화

## 9. 충돌 이슈 처리 내역

- 심현석 담당 섹션 1~12 및 선행작업의존 패치 중 `docs/rw/충돌 이슈.md`에 신규 등록해야 할 충돌 이슈는 발견하지 않았다.
- 기존 김도성 섹션 6~7 미완료로 인한 `bus_info_gateway` 조건부 보류는 선행 섹션 완료 확인 후 패치 완료했다.
- 후속 문서 정합성 검토에서 `DATA_SCHEMA.md` 및 RTDB schema의 낡은 `/busArrivals/{stopId}/{routeId}` 표현을 실제 구현 기준인 `/busArrivals/{stopId}`로 정리했다.
- public_data real provider 활성화는 김도성 모듈의 운영 환경과 서비스키 확보 상태를 별도 확인해야 하며, 이는 충돌 이슈가 아니라 운영 전환 검수 항목이다.
