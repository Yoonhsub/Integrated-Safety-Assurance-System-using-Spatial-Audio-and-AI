### 4.A LiveBusArrivalsProvider 처리 방침 (사용자 명시 요청 3)

본 절은 심현석 검토 결과 사용자 명시 요청에 따라 추가된 운영 인지 사항이다.
4월 MVP 범위에서 `LiveBusArrivalsProvider`(real API 호출 경로)는 **의도적으로 stub 상태**로 유지된다.

**4월 MVP 시점 동작 확인**:

```txt
[mock 흐름 — PUBLIC_DATA_USE_MOCK=true (기본값)]
  - BusArrivalsService(use_mock=True).get_arrivals(stop_id) → MockBusArrivalsProvider 호출
  - mock_bus_arrivals.json 4건 로드 → NormalizedBusArrivalsResponse 반환
  - shared schema 정합성 PASS (top required {stopId, arrivals} + arrival item required + congestion enum 4종 cover)
  - mock 4건 다양성: lowFloor T·F, congestion 4종, remainingStops null·0·정수
  - 환경 제약(컨테이너 네트워크 차단으로 pip install pydantic 불가)으로 Pydantic 인스턴스 통합 smoke만 미실행 — 운영 환경 별도 수행 필요
  - stdlib + Fake 클래스 시뮬레이션 매 섹션 PASS

[live 흐름 — PUBLIC_DATA_USE_MOCK=false]
  - BusArrivalsService(use_mock=False).get_arrivals(stop_id) → LiveBusArrivalsProvider.get_arrivals 호출
  - LiveBusArrivalsProvider.get_arrivals → raise NotImplementedError("LiveBusArrivalsProvider.get_arrivals call body is not implemented yet. Activate by ... Until then, set PUBLIC_DATA_USE_MOCK=true.")
  - 즉 USE_MOCK=false 환경에서 모든 호출이 즉시 실패. silent fallback 위험 0건.
```

**의도적 미구현 사유**:

```txt
- 운영 환경 명세 미확보:
  - TAGO ArvlInfoInqireService 활용신청 명세서 미수령 (cityCode/nodeId 필드 정확한 키명 확인 필요)
  - 서울 BIS getArrInfoByRouteAll의 응답 필드 명세는 1차 출처(http://api.bus.go.kr/contents/sub02/getArrInfoByRouteAll.html)에서 확인 완료, 단 인증키 발급 환경 변동 (국가정보자원관리원 화재 2025-09-26 → 2026-05 시점 부분 정상화)으로 캡스톤 학생 환경에서 발급-승인 시간 비용·1일 1,000건 트래픽 제약·외부 서비스화 별도 절차가 부담
- 4월 mock-first 정책의 합리성 (services/public_data/README.md §6 명시):
  1. 발급 신청 → 승인 → 활용까지의 시간 비용이 4월 MVP 일정과 맞지 않을 수 있음
  2. 자동 단위 테스트가 빈번히 호출하는 패턴에서 1일 1,000건 트래픽 제약 부담
  3. 상업화·외부 서비스화 시점에 별도 운영계정 신청·심의 승인 절차 추가 필요
- 김도성 4월 담당 트리거 프롬프트 §1·§5의 "4월에는 mock-first" 정책 부합
```

**활성화 조건 (운영 환경 명세 확보 시)**:

```txt
1. PUBLIC_DATA_API_KEY 환경변수에 발급 키 설정 (코드/ZIP에 키 포함 절대 금지)
2. (선택) PUBLIC_DATA_CITY_CODE 환경변수에 TAGO 도시코드 설정 (서울 BIS만 쓸 경우 미설정)
3. PUBLIC_DATA_USE_MOCK=false로 mock 모드 해제
4. LiveBusArrivalsProvider.get_arrivals의 raise NotImplementedError(...) 라인 제거
5. _call_arrivals_api의 raise NotImplementedError(...) 가드 제거 + 주석 처리된 boilerplate의 코드 라인 "# " 접두사 풀기 (STEP 박스 헤더 ====[ STEP N ]====는 그대로 유지)
6. 활성화 후 단위 통합 테스트 (실 키 발급 후 1회 호출 → 실 응답 normalize PASS 확인)
```

활성화 절차 1·2·3은 코드 변경 없음 (환경변수만 설정). 활성화 절차 4·5는 김도성 단독 가능 (boilerplate가 이미 작성되어 있어 주석만 풀면 동작). 활성화 절차 6은 심현석 백엔드 통합 테스트와 같이 진행 가능.

**USE_MOCK=false NotImplementedError 발생 인지 여부**: **인지함**. 4월 MVP 의도된 stub이며, 위 활성화 조건이 충족되기 전까지는 USE_MOCK=true(기본값)로 운영. live 경로 활성화는 운영 환경 명세 확보 시점에 김도성 측에서 즉시 가능하도록 boilerplate(STEP 박스 5개 + 활성화 절차 docstring) 준비 완료.
