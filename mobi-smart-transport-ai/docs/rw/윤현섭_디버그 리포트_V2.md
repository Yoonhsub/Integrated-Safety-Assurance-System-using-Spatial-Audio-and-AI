# 윤현섭 V2 디버그 리포트

## 1. 개요

본 문서는 `mobi-smart-transport-ai` 프로젝트의 윤현섭 담당 V2 개발 과정에서 발생한 주요 문제와 해결 과정을 정리한 디버그 리포트이다.

V2 개발은 기존 4월 MVP UI를 실제 API 연동 가능한 구조로 고도화하는 과정이었다. 승객 앱과 기사 앱에서 `BackendApiClient`, API 연결 실패 fallback, STT/TTS, 접근성 Semantics, ride request 흐름을 단계적으로 검증하였다.

수정 및 검증 범위는 다음으로 제한하였다.

- `apps/passenger_app/**`
- `apps/driver_app/**`

다음 범위는 임의 수정하지 않았다.

- `backend/**`
- `backend/api/**`
- `services/public_data/**`
- `packages/shared_contracts/**`
- `packages/mobile_sensors/**`
- `infrastructure/**`
- `ai_vision/**`

## 3. 상세 디버그 기록

### 3.1 BackendApiClient 구조 정리 및 한글 문구 정리

#### 문제

V2 시작 시 passenger_app과 driver_app의 `BackendApiClient` 구조가 서로 다르고, 일부 mock 문구가 깨진 상태였다. 또한 실제 API 연결을 시도하기 위한 `timeout`, `useMockData`, `/health` 구조가 명확하지 않았다.

#### 원인

4월 MVP 단계에서는 mock-first UI 구현이 중심이었기 때문에, 실제 API 연결을 위한 공통 client 기준이 충분히 정리되어 있지 않았다.

#### 해결

다음 내용을 정리하였다.

- passenger_app `BackendApiClient` 정리
- driver_app `BackendApiClient` 정리
- `baseUrl` 유지
- `useMockData` 추가
- `timeout` 추가
- `fetchHealthStatus()` 추가
- mock 문구 한글 정상화

#### 결과

후속 `/health`, bus arrivals, ride requests, driver requests 연동에서 동일한 방식으로 API 연결 및 fallback 처리를 구성할 수 있게 되었다.

---

### 3.2 `/health` 연결 실패 fallback 확인

#### 문제

로컬 백엔드 서버가 실행 중이지 않은 상태에서 `/health` API 연결에 실패하였다.

#### 원인

`MOBI_API_BASE_URL=http://localhost:8000` 기준으로 앱이 `/health` 요청을 보냈으나, 로컬 백엔드 서버가 켜져 있지 않았다.

#### 해결

실패 자체를 오류로 보지 않고 UI fallback으로 처리하였다.

확인된 fallback 문구는 다음과 같다.

- 백엔드 연결 상태
- 연결 실패
- 백엔드 `/health` API에 연결할 수 없습니다.

#### 결과

백엔드 미실행 상태에서도 앱이 깨지지 않고 연결 실패 상태를 정상적으로 표시하였다.

---

### 3.3 버스 도착 정보 카드가 기존 mock snapshot을 계속 사용하던 문제

#### 문제

Section 03에서 `fetchBusArrivalSummary()`를 추가했지만, 실행 화면에서는 버스 도착 정보 카드가 계속 기존 mock 문구를 표시하였다.

#### 원인

`BackendApiClient`에는 새 버스 도착 정보 요청 구조를 추가했으나, `home_page.dart`의 버스 도착 정보 카드가 여전히 `_homeSnapshot?.busArrivalStatus` 값을 사용하고 있었다.

#### 해결

버스 도착 정보 카드를 `_busArrivalSummary` 기준으로 렌더링하도록 수정하였다.

수정 방향은 다음과 같다.

- 기존: `_homeSnapshot?.busArrivalStatus`
- 수정: `_busArrivalSummary`
- loading 기준: `_isLoadingBusArrivals`

#### 결과

백엔드가 꺼진 상태에서 다음 fallback이 정상 표시되었다.

- 버스 도착 정보
- 연결 실패
- 버스 도착 정보 API에 연결할 수 없습니다.

### 3.4 `BusArrivalSummary` 클래스 위치 문제

#### 문제

`BusArrivalSummary` 클래스를 추가한 뒤 Dart 구조 오류가 발생하였다.

#### 원인

`BusArrivalSummary` 클래스를 `BackendHealthStatus` 클래스 내부에 넣어버린 것이 원인이었다. Dart에서는 클래스 안에 다른 클래스를 선언할 수 없다.

#### 해결

`BusArrivalSummary`를 `BackendHealthStatus` 바깥으로 분리하였다.

정상 구조는 다음과 같다.

- `BackendHealthStatus`
- `BusArrivalSummary`
- `PassengerHomeSnapshot`
- `PassengerStatusItem`

#### 결과

`BackendApiClient`에서 `BusArrivalSummary`를 정상적으로 참조할 수 있게 되었고, `flutter analyze` 문제가 해결되었다.

---

### 3.5 `_createRideRequest` 미참조 warning

#### 문제

Section 06 검증 중 다음 warning이 발생하였다.

- `The declaration '_createRideRequest' isn't referenced.`

#### 원인

`_createRideRequest()` 메서드는 존재했지만, 실제 화면의 버튼 `onPressed`에 연결되어 있지 않았다.

#### 해결

“탑승 요청하기” 버튼을 추가하고, `onPressed`를 `_createRideRequest`에 연결하였다.

적용한 연결 방식은 다음과 같다.

- `onPressed: _isCreatingRideRequest ? null : _createRideRequest`

#### 결과

unused warning이 사라졌고, 탑승 요청 생성 흐름이 UI 버튼과 연결되었다.

### 3.6 탑승 요청하기 버튼 누락 문제

#### 문제

상태 조회 버튼은 있었지만 “탑승 요청하기” 버튼이 화면에서 누락되어 있었다.

#### 원인

Section 06 상태 조회 버튼을 추가하는 과정에서 탑승 요청 생성 버튼이 빠진 상태로 남아 있었다.

#### 해결

탑승 요청 상태 카드 아래에 다음 순서로 버튼을 배치하였다.

1. 탑승 요청 상태 카드
2. 탑승 요청하기 버튼
3. 상태 조회 버튼
4. MVP 안내 카드

#### 결과

탑승 요청 생성과 상태 조회를 모두 화면에서 실행할 수 있는 구조가 되었다.

### 3.7 `fetchRideRequestStatus` 위치 문제

#### 문제

`home_page.dart`에서 다음 호출부에 빨간 줄이 발생하였다.

- `_backendApiClient.fetchRideRequestStatus(...)`

#### 원인

`fetchRideRequestStatus()` 메서드가 `BackendApiClient` 클래스 안에 들어가지 않았거나, 중괄호 위치 문제로 클래스 밖에 위치한 상태였다.

#### 해결

`fetchRideRequestStatus()`를 `BackendApiClient` 클래스 안에 배치하였다.

정상 구조는 다음과 같다.

- `fetchHealthStatus()`
- `fetchPassengerHomeSnapshot()`
- `fetchBusArrivalSummary()`
- `createRideRequest()`
- `fetchRideRequestStatus()`
- `_mockBusArrivalSummary()`
- `_buildUri()`

#### 결과

`home_page.dart`에서 `_backendApiClient.fetchRideRequestStatus()`를 정상적으로 호출할 수 있게 되었다.

### 3.8 `children` 리스트 안에 중괄호 사용으로 인한 `Set<InvalidType>` 오류

#### 문제

버튼 추가 후 다수의 Flutter 분석 오류가 발생하였다.

대표 오류는 다음과 같다.

- `The element type 'Set<InvalidType>' can't be assigned to the list type 'Widget'.`

#### 원인

`children: [ ... ]` 위젯 리스트 안에 불필요한 `{ ... }` 중괄호 블록을 넣었고, Dart가 이를 Set으로 해석하였다.

잘못된 형태는 다음과 같다.

- `children: [ { const SizedBox(...), Semantics(...) } ]`

#### 해결

중괄호를 제거하고 위젯을 리스트 안에 그대로 나열하였다.

정상 형태는 다음과 같다.

- `children: [ const SizedBox(...), Semantics(...) ]`

#### 결과

다수의 `Set<InvalidType>` 오류가 해결되었고, `flutter analyze`가 정상 통과하였다.

---

### 3.9 driver_app 요청 수락 버튼 검증 한계

#### 문제

Section 08에서 “요청 수락” 버튼을 추가했지만, 백엔드가 꺼져 있거나 요청 목록이 비어 있으면 버튼이 화면에 표시되지 않는다.

#### 원인

요청 수락 버튼은 `GET /drivers/{driverId}/ride-requests` 응답 목록이 있어야 표시된다. 로컬 백엔드 서버가 실행 중이지 않은 상태에서는 요청 목록 조회가 실패 fallback으로 처리되기 때문에 실제 버튼 동작까지 검증하기 어렵다.

#### 해결

검증 범위를 다음과 같이 분리하였다.

백엔드 미실행 상태에서 확인한 내용은 다음과 같다.

- `flutter analyze` 확인
- 요청 목록 조회 실패 fallback 확인
- 앱 화면 깨짐 없음 확인

백엔드 및 테스트 데이터가 존재할 때 추가로 검증해야 할 내용은 다음과 같다.

- 요청 카드 표시 확인
- 요청 수락 버튼 표시 확인
- `PATCH /ride-requests/{requestId}/status` 호출 확인
- `ACCEPTED` 상태 변경 성공 확인

#### 결과

현재 V2에서는 백엔드 미실행 상태 fallback과 코드 연결 검증을 완료하였다. 실제 상태 변경 성공 검증은 후속 통합 단계에서 진행해야 한다.

### 3.10 geofence/mobile_sensors 담당 범위 문제

#### 문제

Section 11에서 안전 상태 카드를 실제 geofence/sensor 이벤트와 연결할 수 있는지 검토하였다.

#### 확인 결과

백엔드에는 다음 구조가 존재하였다.

- `POST /geofence/check`
- `GeofenceCheckRequest`
- `GeofenceCheckResponse`
- `GeofenceStatus`
- `SAFETY_ALERT`

또한 `packages/mobile_sensors`에는 스마트폰 방향/나침반 센서, BLE, RSSI 기반 거리 추정 구조가 존재하였다.

#### 판단

`packages/mobile_sensors/**`, `backend/**`, `shared_contracts/**`는 윤현섭 담당 범위 밖이다. 따라서 실제 센서 연결이나 geofence 요청 body 확정은 이번 범위에서 직접 구현하지 않았다.

#### 해결

Section 11은 코드 수정 없이 다음 내용을 검증 완료 처리하였다.

- geofence API 존재 확인
- safety alert 흐름 확인
- mobile_sensors skeleton 단계 확인
- passenger_app 안전 상태 카드 placeholder 확인
- Semantics/fallback 구조 확인

#### 결과

담당 범위를 벗어나지 않고 안전 상태 표시 구조 검증을 완료하였다.

## 6. 결론

V2 개발 과정에서 발생한 주요 문제는 대부분 Flutter 위젯 배치, Dart 클래스 위치, 버튼 연결, API fallback 처리와 관련된 문제였다. 각 문제는 윤현섭 담당 범위인 `apps/passenger_app/**`, `apps/driver_app/**` 내부에서 해결하였다.

백엔드, shared contracts, mobile_sensors 등 범위 밖 파일은 임의 수정하지 않았으며, 실제 연동이 필요한 부분은 후속 통합 단계로 분리하였다.

따라서 V2 디버그 기준에서 윤현섭 담당 앱 영역은 안정적으로 정리되었다고 판단한다.
