# 윤현섭 파트 졸업작품 인터뷰 예상 질문 정리

## 1. 인터뷰 준비 핵심 요약

### 본인 담당 범위

윤현섭 파트는 Flutter 기반 앱 UI를 담당하였다.

- `apps/passenger_app/**`
- `apps/driver_app/**`

주요 구현 내용은 다음과 같다.

- 승객 앱 홈 화면 UI
- 큰 음성 입력 버튼
- Voice-First 흐름
- STT/TTS wrapper 및 실제 plugin 연동 보강
- 접근성 `Semantics label/hint`
- 승객 앱 안전 상태, 버스 도착 정보, 탑승 요청 상태 mock 렌더링
- 기사 앱 탑승 요청 목록 mock UI
- 기사 앱 요청 확인 상태 변경 흐름
- 최종 개발 보고서 및 디버그 리포트 작성

### 지켜야 할 답변 원칙

인터뷰에서 가장 중요한 답변 원칙은 다음과 같다.

- 내가 직접 구현한 부분과 후속 작업을 구분해서 말한다.
- 백엔드/API/schema/status enum은 임의 확정하지 않았다고 설명한다.
- mock/placeholder를 쓴 이유를 협업 충돌 방지 관점에서 설명한다.
- 접근성, STT/TTS, mock 렌더링, 기사 앱 요청 목록을 핵심 키워드로 말한다.
- 실제 서비스화를 위해 필요한 후속 작업을 명확히 말한다.

## 2. 가장 가능성 높은 질문 7개

### Q1. 본인이 맡은 역할이 정확히 무엇인가요?

저는 윤현섭 파트로서 `apps/passenger_app`과 `apps/driver_app`의 Flutter UI를 담당했습니다.  
승객 앱의 음성 입력 UI, 상태 카드, 접근성 UI, STT/TTS 흐름, 그리고 기사 앱의 탑승 요청 화면 초안을 구현했습니다.

### Q2. 왜 mock이나 placeholder를 사용했나요?

백엔드 API와 schema는 다른 팀원 담당 영역이라 제가 임의로 확정하면 충돌이 생길 수 있었습니다.  
그래서 앱 내부에서는 mock 렌더링 구조와 UI shell을 만들고, 실제 API 계약이 확정되면 쉽게 교체할 수 있도록 구성했습니다.

### Q3. STT/TTS는 실제로 동작하나요?

초기에는 wrapper stub 구조로 시작했지만, 피드백 이후 `speech_to_text`와 `flutter_tts`를 연결하여 실제 음성 인식과 안내 출력이 가능한 구조로 보강했습니다.  
다만 실기기 권한, 브라우저별 동작 차이, 음성 인식 실패 처리 등은 후속 검증이 필요합니다.

### Q4. 시각장애인과 노약자를 위해 어떤 접근성 처리를 했나요?

큰 음성 입력 버튼, 명확한 상태 카드, 상태 배지, `Semantics label/hint`, 음성 안내 흐름을 적용했습니다.  
시각장애인이나 노약자가 작은 UI를 찾지 않아도 핵심 기능을 쉽게 인식하고 사용할 수 있도록 구성했습니다.

### Q5. 기사 앱의 탑승 요청은 실제 API인가요?

현재는 실제 `GET /drivers/{driverId}/ride-requests` API를 호출하지 않습니다.  
대신 최소 1건 이상의 mock 탑승 요청 목록을 표시하고, 요청 확인 버튼을 누르면 UI 상태가 “확인 대기”에서 “확인 완료”로 바뀌도록 구현했습니다.

### Q6. 왜 백엔드나 shared contracts를 수정하지 않았나요?

프로젝트 규칙상 제 담당 범위는 `apps/passenger_app/**`, `apps/driver_app/**`입니다.  
공통 계약, API schema, DB schema, status enum은 다른 팀원 영역과 연결되므로 임의로 수정하지 않았습니다.

### Q7. 실제 서비스로 만들려면 다음 단계는 무엇인가요?

실제 geofence API, 버스 도착 정보 API, rideRequests API 연동이 필요합니다.  
또한 Firebase/FCM 알림, 실기기 접근성 테스트, 네트워크 예외 처리, 실제 사용자 테스트가 필요합니다.

## 3. 프로젝트 전체 이해도 질문

### Q8. 이 프로젝트의 핵심 목적은 무엇인가요?

시각장애인과 노약자가 대중교통을 더 안전하고 쉽게 이용할 수 있도록 돕는 스마트 교통 AI 에이전트 시스템입니다.  
승객 앱은 음성 기반 입력과 안내를 제공하고, 기사 앱은 탑승 요청 확인 흐름을 담당합니다.

### Q9. 기존 버스 앱과 차별점은 무엇인가요?

일반 버스 앱은 주로 도착 정보 제공에 집중합니다.  
이 프로젝트는 접근성, 음성 입력, 안전 상태 안내, 기사에게 탑승 요청 전달까지 포함한다는 점이 다릅니다.

### Q10. 왜 시각장애인과 노약자를 주요 사용자로 설정했나요?

정류장 위치 확인, 버스 도착 판단, 차량 탑승 여부 판단이 일반 사용자보다 어렵기 때문입니다.  
그래서 음성 안내, 큰 버튼, 기사 앱 연계가 필요하다고 판단했습니다.

### Q11. AI 에이전트 시스템에서 본인 앱 UI 파트의 역할은 무엇인가요?

AI나 백엔드 모듈에서 나온 정보를 사용자가 이해할 수 있도록 전달하는 앱 계층입니다.  
제 파트는 AI 모델 자체보다는 접근성 있는 UI와 상태 전달 구조에 초점을 두었습니다.

## 4. 본인 담당 범위 질문

### Q12. 본인이 직접 수정한 주요 파일은 무엇인가요?

주요 수정 파일은 다음과 같습니다.

- `apps/passenger_app/lib/src/pages/home_page.dart`
- `apps/passenger_app/lib/src/services/voice_guide_service.dart`
- `apps/passenger_app/lib/src/services/backend_api_client.dart`
- `apps/driver_app/lib/src/pages/home_page.dart`
- `docs/rw/윤현섭_최종_개발_보고서.md`
- `docs/rw/윤현섭_디버그_리포트.md`

### Q13. 윤현섭 담당 범위 밖 파일은 왜 건드리지 않았나요?

`backend/api`, `infrastructure/firebase`, `services/public_data`, `packages/shared_contracts` 등은 다른 팀원 담당 영역입니다.  
API 계약이나 DB schema를 임의 확정하지 않는 것이 협업 충돌을 줄이는 방식입니다.

### Q14. Section 01~12 방식으로 진행한 이유는 무엇인가요?

필독 문서에서 정한 작업 절차를 따랐습니다.  
짝수 섹션은 구현, 홀수 섹션은 검토/패치 성격으로 진행했고, 마지막 Section 12에서 전체 검증을 수행했습니다.

## 5. 승객 앱 관련 질문

### Q15. 승객 앱에서 가장 중요한 UI는 무엇인가요?

큰 음성 입력 버튼입니다.  
시각장애인과 노약자가 쉽게 찾고 조작할 수 있도록 홈 화면 중심에 배치했습니다.

### Q16. 음성 입력 버튼을 누르면 어떤 흐름이 발생하나요?

버튼 문구가 “음성으로 목적지 입력”에서 “음성 입력 종료”로 바뀝니다.  
음성 안내 상태 카드가 업데이트되고, STT 결과가 있으면 인식 중인 문구가 표시됩니다.  
TTS 안내 문구도 출력되도록 구성했습니다.

### Q17. VoiceGuideService를 분리한 이유는 무엇인가요?

UI와 음성 처리 로직을 분리해 유지보수를 쉽게 하기 위해서입니다.  
실제 STT/TTS 플러그인이 바뀌거나 예외 처리가 추가되어도 UI 전체를 크게 수정하지 않도록 했습니다.

### Q18. BackendApiClient는 어떤 역할인가요?

현재는 실제 API 대신 승객 앱 홈 화면에 필요한 안전 상태, 버스 도착 정보, 탑승 요청 상태 mock snapshot을 반환합니다.  
실제 API 계약이 확정되면 이 service를 실제 HTTP 호출로 교체하면 됩니다.

### Q19. mock과 placeholder의 차이는 무엇인가요?

placeholder는 “나중에 데이터가 들어올 자리”라는 안내에 가깝습니다.  
mock은 실제 데이터처럼 보이는 임시 값을 화면에 렌더링하는 것입니다.  
이번 피드백 반영으로 고정 placeholder에서 mock 렌더링 구조로 개선했습니다.

## 6. 기사 앱 관련 질문

### Q20. 기사 앱은 어떤 역할을 하나요?

기사 앱은 버스 기사가 승객의 탑승 요청을 확인할 수 있도록 하는 화면입니다.  
현재는 mock 탑승 요청 목록과 요청 확인 UI를 제공합니다.

### Q21. 기사 앱에서 실제 rideRequests API를 호출하나요?

아직 호출하지 않습니다.  
실제 `GET /drivers/{driverId}/ride-requests` API와 schema가 확정되면 백엔드 API와 연결해야 합니다.

### Q22. 요청 확인 버튼은 실제 서버 상태를 바꾸나요?

아닙니다.  
현재는 mock UI 상태만 “확인 대기”에서 “확인 완료”로 바꿉니다.  
실제 서버 반영은 rideRequests 파이프라인 확정 후 구현해야 합니다.

### Q23. requestId, userId, busNo 같은 필드를 왜 만들지 않았나요?

그 필드들은 API/schema 계약에 해당합니다.  
제 담당 범위에서 임의 확정하면 충돌 가능성이 있으므로 사람이 이해할 수 있는 mock 문구만 사용했습니다.

## 7. 접근성 관련 질문

### Q24. 접근성을 위해 구체적으로 무엇을 했나요?

Flutter `Semantics`를 사용해 주요 버튼과 상태 카드에 `label`과 `hint`를 넣었습니다.  
또한 큰 버튼, 큰 텍스트, 명확한 상태 배지, 단순한 카드형 레이아웃을 적용했습니다.

### Q25. Semantics label과 hint의 차이는 무엇인가요?

`label`은 스크린리더가 핵심 내용을 읽는 값입니다.  
`hint`는 사용자가 해당 요소를 어떻게 이해하거나 사용할지 보조 설명을 주는 값입니다.

### Q26. 실제 TalkBack이나 VoiceOver 테스트를 했나요?

현재는 Flutter Semantics 적용, Chrome 실행, `flutter analyze` 검증까지 했습니다.  
실제 Android TalkBack과 iOS VoiceOver 실기기 테스트는 후속 작업입니다.

## 8. API, mock, FCM 관련 질문

### Q27. 실제 API가 완성되면 어떤 부분을 바꾸면 되나요?

승객 앱은 `BackendApiClient`의 mock 반환 부분을 실제 HTTP 호출로 교체하면 됩니다.  
기사 앱은 mock 목록을 실제 rideRequests API 응답 렌더링으로 교체하면 됩니다.

### Q28. FCM 토큰 등록은 왜 구현하지 않았나요?

FCM 토큰 등록은 Firebase 초기화, `firebase_messaging`, 플랫폼별 설정, 서버 토큰 등록 API 계약이 필요합니다.  
이는 윤현섭 앱 UI 범위를 넘어 Firebase/백엔드 담당 영역과 연결되므로 후속 작업으로 문서에 명시했습니다.

### Q29. 알림 기능은 앞으로 어떻게 연결할 수 있나요?

Firebase/FCM 설정과 서버 토큰 등록 API가 확정되면, 기사 앱은 탑승 요청 알림을 받고 해당 요청을 목록에 표시하는 흐름으로 확장할 수 있습니다.

## 9. 문서 및 검증 관련 질문

### Q30. 최종 개발 보고서에는 무엇을 작성했나요?

담당 범위, Section 01~12 수행 결과, 구현 완료 기능, 수정 파일, 테스트 결과, 미구현 기능, 한계 사항, 충돌 이슈, 최종 판정을 정리했습니다.

### Q31. 디버그 리포트에는 무엇을 작성했나요?

섹션별 확인 항목, 발생 문제, 해결 내용, 테스트 결과, 남은 mock/placeholder, 실제 테스트하지 못한 항목을 정리했습니다.

### Q32. 테스트는 어떻게 했나요?

`passenger_app`과 `driver_app` 각각에서 `flutter analyze`를 실행했고, Chrome으로 UI 실행을 확인했습니다.  
버튼 토글, 상태 카드 표시, mock 렌더링, MVP 안내 카드 표시를 확인했습니다.

### Q33. 자동 테스트는 왜 없나요?

이번 범위는 UI shell과 접근성 렌더링 중심이어서 수동 실행과 `flutter analyze` 중심으로 검증했습니다.  
후속으로 위젯 테스트를 추가할 수 있습니다.

## 10. 인터뷰 답변용 핵심 문장

### 담당 범위

저는 윤현섭 파트로서 `apps/passenger_app`과 `apps/driver_app`의 Flutter UI를 담당했습니다.

### mock 사용 이유

API와 schema는 다른 팀원 담당 영역이므로 임의 확정하지 않고 mock 렌더링과 UI shell 중심으로 구현했습니다.

### STT/TTS

초기 stub에서 시작했지만 `speech_to_text`와 `flutter_tts` 기반 plugin wrapper 구조로 보강했습니다.

### 접근성

시각장애인과 노약자를 고려해 큰 버튼, 상태 카드, `Semantics label/hint`, 음성 안내 흐름을 적용했습니다.

### 한계점

실제 백엔드 API, FCM, rideRequests, 실기기 접근성 테스트는 후속 작업입니다.

## 11. 마무리 조언

- 답변할 때 “제가 구현한 것”과 “후속 작업”을 명확히 구분하면 좋습니다.
- 백엔드/API/schema를 임의 확정하지 않았다는 점은 방어적으로 설명할 수 있는 중요한 포인트입니다.
- 접근성, STT/TTS, mock 렌더링, 기사 앱 mock 요청 목록은 본인 파트의 핵심 키워드입니다.
- 모르는 질문이 나오면 “해당 부분은 담당 범위 밖이지만, 제 파트에서는 연동 가능한 UI/service 구조를 준비했다”라고 답변하면 안정적입니다.
